ISMS Clause / Control: A 14.2.1 ICPo15 Version: 1.00 Date: 10th April, 2017

15.0 Purpose and Scope

The purpose of this documented information is to provide an overview of the various policies. The required policies to be adhered in all processes preserve the Confidentiality, Integrity and Availability of the information system of the organization. Applicable to all activities, personnel performing the activities, personnel responsible for the activity to be performed in the defined manner covering the scope of activities at organization.

15.4. Procedure

15.4.1. General

The policy for information security defined below has been approved by management for publication and communication to employees along with relevant external parties.

15.4.2. Secure Development Policy

15.4.2.1. Development of code shall be checked and validated with the most current versions of Coding Standards for Secure Application Development.

15.4.2.2. All code developers shall verify that their code is in compliance with the most recent and approved coding standards and guideline.

15.4.2.3. Only validated code shall be implemented into the production environment.

15.4.2.4. Ensure code meets the level of confidence that software is free from exploitable code vulnerabilities, regardless of whether they are already designed into the software or inserted later in its life cycle.

15.4.2.5. Ensure code provides predictable execution or justifiable confidence and that the software, when executed, will provide security functionality as intended.

15.4.2.6. Never trust incoming data to the system, apply checks to this data.

15.4.2.7. Error messages should not reveal any security information to the user.

15.4.2.8. System/Application logs should not have security information like password, username, security tokens etc.

15.4.2.9. Use environment variables prudently and always check boundaries and buffers.

15.4.2.10. Ensure the Procedure for software development is followed in the organization.

15.4.2.11. Development, testing, and operational environments must be separated to reduce the risks of unauthorized access or changes to the operational environment.
