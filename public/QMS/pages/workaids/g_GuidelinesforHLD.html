1.0    Overview and Objectives

The SDD should be modular; that is the software is divided into separately named and addressable components that are integrated to satisfy the entire set of software requirements. SDD also contains the internal details of each component and describes the inputs, outputs, local data, global data, and the algorithm for each component. The programmers prior to coding need these details, for developing the components. The design could be broadly divided into following types:

�         GUI Design focuses on the user interface.

�         Architectural Design involves breaking of the system into components. This process of analyzing and identification of components is termed as decomposition. The components could be a program, sub-routine or a function depending on the degree of decomposition. The decomposition should be such that each component addresses a specific sub-function of the requirement and has a simple interface.

�         Data Design involves identifying the data elements and defining the data structures appropriate to hold the data elements. It may also specify the data storage (e.g. memory), and the routines that are required to access them.

�         Logic Design (also called program specification) focuses on processing details of each component individually. The processing details include the sequence of events, repetitive operations, decision points and data organization.

The main objectives of the SDD document are:

�         To identify the framework based on which the system will be built

�         To identify the components and their interfaces

�         To identify common/reusable components

�         To form the basis for integration test plan

�         To form the basis for Unit Test Plan
