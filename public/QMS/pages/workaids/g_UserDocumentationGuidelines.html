1.   Designing the Blueprint for User Documentation

A software product may need more than one type of documentation. In such cases, the planning occurs in two phases. First, the library blue print, containing a design meant for all the documents, and then the individual document blueprint.

The blueprint is prepared and presented to either the management or clients or both. This is to make the management and clients aware of the documentation plans. This blueprint must get the approval of the person authorizing and reviewing the final document.

The blue print should cover all aspects of documentation, including the concept of audience, purpose, content, graphics, reference aids, format, updating and production cycle. It should also include the layout and organization, content, style and plans for production, update and review.

Any competent writer should be able to successfully finish writing the manual by following the blueprint without asking any more questions.

1. Text Organization

1. Stick to the expectation of general readers

2. Make the organization of the material apparent

3. Arrange the text in

·         Chronological order

·         Order of importance

·         Order of need

·         Order of difficulty

4. Develop product visualization; a picture should emerge in the reader’s mind

5. Include related information

6. Generate examples

2. Questions to Ask

In developing task-oriented documentation, the following questions need to be asked for each task.

·         Who performs the task?

·         What action begins the task?

·         Are there any prerequisites for performing the task?

·         What are the specific steps involved in performing the task?

·         What action ends the task?

·         Does the result of the task depend upon any variations in any other conditions?

Documents must have excellent quick reference capabilities.

2.   Information Procurement

There are basically four methods of procuring the material needed to prepare a software manual

1.      External specifications

The external specifications written at the design level of the software is the most important information to start with. The specifications describe the product, its operation and use. Note down all the unfamiliar references and terms, look up other manuals and you may have to duplicate the information in your manual, as anything that is unfamiliar to you might be unfamiliar to most of the readers. If there is something unfamiliar still, do not track down the author and start asking questions: note the questions down

2.      Informal programming notes

When programmers are coding, informal notes are made. Ask the programmers to give a copy of all the notes made. They prefer this to explaining things to you while you are hanging around. However, do not return to them with these notes and start asking questions; note the questions down.

3.      Program listings

Try to get hold of a copy of the listings. Again, do not return to them with these listings and start asking questions; note the questions down

4.      Ask questions

Questions have a tendency to answer themselves when you finish read through all your sources of information. You will be able to cross off many questions by this time. Ask questions after organizing all the questions and gaining some knowledge of the product. 

3.   Formatting Guidelines

1.     The same template (manuals.dot) should be attached to all the user manuals before the drafting of a manual starts. This template should be copied on all the systems of the documentation group, in the same path. The template can be changed if the customer wants it differently.

2.     The page setup for user manuals is stored in the template file as text. The same page setup should be used.

3.     Section breaks should be given at the end of each chapter with a new section starting at odd page. Printing of the pages is done with mirror margins.

4.     The first page of a chapter will not have a header. For the rest of the pages, the right side pages will have the chapter name in the header and the left side pages will have the name of the product.

5.     Page numbers in the footer should be inserted with mirroring. The right side pages will have "Integra Micro Systems (P) Ltd." in the left part of the footer and the page number in the right part. The left side pages will  have only the page number in the left part of the footer.

4.    Printing and Production

·         All the user manuals should be printed after the development is complete.

·         One master copy should be preserved by the Documentation Group and one copy should be provided to the Production Department and used for making copies.

·         The date of printing should be printed on the bottom-right side of the backside cover page.

·         The version number should be printed on the cover page along with the name of the product

·         Cover page, disclaimer and copyrights notice should appear in the standard format provided. Please refer to Coverdisc

·         Whenever an update is made, the revised version should have the version number and date of printing updated.

·         Whenever an update is made, the Production Department should be notified and a copy of the revised version should be sent to the Production Department.

These guidelines refer to the manuals produced for Integra’s products. For the manuals developed for the products developed by SSSD as a software development project or for the manuals developed as a project for a client by the Technical Documentation Group guidelines given by the customer should be used.

5.    Creating Online Help

1.     Design a blueprint for the online help and submit for approval.

2.     Use help.dot template while creating help.

3.     Provide browse buttons and create context sensitive browse sequences

4.     Do not include bmp files in the rtf file. Give links to the bmp files.

5.     Include hyper graphics whenever required.

6.     Generally, try to restrict topic size to not more than one screen length. Create hotspots.

7.     Break-up the rtf file into multiple files of logical (functionality) or physical units (menu options) whenever the software is large.

8.     Keep all the image files (screens etc.) in a separate directory inside the root directory of the help files and include the bitmap directory in the project file.

9.     Provide full text search.

10.  If the user manual has been already prepared, use the material from the manual.
