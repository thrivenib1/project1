Coding Standard for C Language
CS-02 Version: 3.00 Date: 17th May 2010

2.0 Introduction

2.1 Purpose

The purpose of this Standard is to provide coding standard for C language.

2.2 Scope

This Standard is applicable for developing C language based applications.

2.3  File Organization

2.3.1 File Contents

�         Files should be used to organize related code modules, at a function level..

�         Any structure must be in the header file (.h). Function prototypes must be used, and will be in the header file also. Function definitions will be in the implementation file (.c).

2.3.2   Source File Layout

�         Source files should contain the following components in the order shown below:

�         A prologue that tells what is in the file.

�         Any Header files.

�         Any defines and typedefs.

�         External data declarations (externs, non-static globals, static globals).

�         Functions (Should be in some sort of meaningful order).

�         Global declarations should begin in column 1.

�         All external data declaration should be preceded by the extern keyword.

�         If an external variable is an array that is defined with an explicit size, then the array bounds must be repeated in the extern declaration unless the size is always encoded in the array.

�         Unrelated declarations, even of the same type, should be on separate lines.

�         For structure and union template declarations, each element should be in a separate line. The closing brace (}) should be in column 1 of subsequent lines

�         When it's possible to put a needed #include line in the source file instead of in the header file, do so. This will reduce unnecessary file dependencies and save a little compile time.

2.3.3     Header File Layout

�         Header files should contain the following components in the order shown below:

�         File Guard

�         System # includes

�         Application # includes

�         #defines

�         Macros

�         Constant

�         External Functions

�         External variables

�         Structs

2.4     Function Declarations

Header File Guard

All header files should contain a file guard mechanism to prevent multiple inclusion. This mechanism is implemented as shown by the following lines:

#ifndef Meaningful NameH /* first line of the header file */

#define Meaningful NameH /* second line of the header file */

/* body of the header file */

#endif /* last line of the header file; note comment */

2.5   Naming Conventions

#define constants should be all CAPS.

Enum constants are Capitalized or in all CAPS.

Function, typedef, and variable names, as well as struct, union, and enum tag names should be in lower case.

Names should give some indication of its use in a program.

Avoid names that differ only in case. Similarly, avoid names that look like each other.

Prefixes and suffixes in variable names and constants should be separated by "_".

2.6 Style Guidelines

Line Length

All lines should be displayable without wrapping on a 72-character display. If wrapping is required, try to break at an operator, and start the next line with the operator vertically aligned.

Example:

printf("I am an Integran ");

Statements Per Line

There should be only one statement per line unless the statements are very closely related.

The null body of a "for" or "while" loop should be on a separate line and commented.

When a block of code has several labels, the labels are placed on separate lines.

2.7 Comments

All comments must precede the actual program code described, except for the line comments on the same line as that of the code they describe.

The comment should describe what is happening, how it is being done, what parameters mean, which globals are used and which are modified, and any restrictions or bugs.

Comments that describe data structures, algorithms, etc., should be in block comment form with the opening /* in columns 1-2, a * in column 2 before each line of comment text, and closing */ in columns 2-3. An alternative is to have ** in columns 1-2, and put the closing */ also in 1-2.

Example:

/*

Here is a block comment.

The comment text should be creating using tab or spaced over uniformly.

The opening slash-star and closing star-slash are alone on a line.

*/

/*

** Alternate format for block comments

*/

Block comments should be aligned to the code that they describe.

Very short comments may appear on the same line as the code they describe, and should provide a tab space to separate them from the statements.

Comments should be placed at the same indentation level as that of the code they describe.

Code, which is machine dependent, should be commented to facilitate its portability.

Deliberate fall-through in switch statements should be commented.

2.8 Blank Lines

If it makes the code more readable, use a single blank line to separate logical groups of code to improve readability. In source files, use two blank lines to separate each function definition, or in some way make it easy to tell where the new function begins.

2.9 Indentation and Braces

The contents of all code blocks should be indented to improve readability. A single tab or four spaces are recommended as the standard indentation. Braces should be placed to show the level of indentation of the code block. Make it as readable as possible.

int main()

{

doSomething()

}

struct myStruct

{

int x;

int y;

}

if ( value == 0 )

{

doSomething();

}

else if ( value == 2 ) /* note position of cascaded if statement */

{

doSomething2();

}

else

{

doSomething3();

}

while ( value < 300 )

{

doSomething();

}

do

{

doSomething();

} while ( value < 300 ) /* ending brace and control on same line*/

switch ( value )

{

case 1:

doSomething();

break;

case 2:

case 3:

doSomething2();

break;

default:

break;

}

2.10 Programming Practices

Hard-coding should be avoided.

Do not check a boolean value for equality with 1 (TRUE, YES, etc); instead test for inequality with 0 (FALSE, NO, etc.)

2.11 Switch Statements

When a block of code has several labels (unless there are a lot of them), the labels are placed on separate lines. The fall-through feature of the C switch statement, (that is, when there is no break between a code segment and the next case statement) must be commented for future maintenance. Switch statements should always have a default case.

Example:

switch (expr)

{

case ABC:

statement;

case DEF:

statement;

break;

case UVW:

statement;

/*FALLTHROUGH*/

case XYZ:

statement;

break;

default:

printf("Unknown Case Encountered");

break

}

2.12 Return Statements

Where practical, have only one return from a function or method as the last statement. Otherwise, minimize the number of returns. Possibly highlight returns with comments and/or blank lines to keep them from being lost in other code. Multiple returns are generally not needed except for error or exception handling.

2.13 Casts

Avoid the use of explicit (older style) casts except where unavoidable, since this can introduce run-time bugs by defeating compiler type-checking. Working with third-party libraries (e.g., X or Motif) often requires the use of casts. When you need to cast, comment the reasons.

2.14 Use constants instead of literal values wherever reasonable

For example:

const double PI = 3.141592; // right

const char* APP_NAME = "ACME Spreadsheet 1.0"; // right

area = 3.141592 * radius * radius; // not recommended

printf("ACME Spreadsheet 1.0"); // not recommended

2.15 Explicit Initialization

In general, explicitly initialize all variables before use.

2.16 Debug Compile-time Switch

Code used only during development for debugging or performance monitoring should be conditionally compiled using #ifdef compile-time switches. These should not be present in the include files (because it is too easy to have a library get out of synch' with the programs that use it).

2.17 Goto and Labels

Avoid using "goto" and "label" statements.
