1.   Overview and Objectives

The Requirement Engineering document describes the process for gathering requirements for the system. In order to proceed with the software development, the requirements stated in the proposal or the URS document needs to be analyzed in detail prior to its translation into software design.

2.   Introduction to Software Requirements

2.1     Definition of Software Requirements

The first step in any software developmental effort is to determine exactly what the software system shall do. Software Requirements stipulates what must be accomplished, transformed, produced or provided. Additionally, a Software Requirement is a software capability that must be met or possessed by a software component in order to satisfy a contract, standard, or specification. The source of these Software Requirements could come from buyers, operators, users, or system specifications. Software requirements can be categorized as Needs, Wishes, and Expectations:

�         Needs are the capabilities and characteristics required in the software system to solve the problem.

�         Wishes are the capabilities and characteristics of the software system desired by the users.

�         Expectations are the capabilities and characteristics of the software system expected by the users.

2.2     The role of Software Requirements

Software Requirements serve two major roles in a developmental effort. They describe what to develop and describe when the development is completed.

A requirements document contains all the Software Requirements of the system that is to be developed. It communicates the customer's needs, wishes, and expectations to the developers of the system and may include descriptions of:

�         The required computational functionality.

�         A description of the system behavior

�         Guidelines for the user interface.

�         Technical aspects of the Hardware / Software Interface

�         Operational characteristics

The second role of Software Requirements is to form the basis for determining when the software product is finished. The verification of the software functionality against the requirement document serves to demonstrate that the contractual agreement between the customer and developers has been met and generally implies the end of the development phase. This verification may require the construction of test situations to objectively prove the software products compliance with the requirements document.

2.3     Types of Software Requirements

There are many different classes of software requirements that are used in defining requirement specification. These classes can be treated as specified characteristics of the requirements rather than different types of requirements. The following are some of the more common requirement classes:

2.3.1     Functional Requirements

The functional requirements define the fundamental actions that the software must perform. The actions describe accepting inputs, processing, and generating the outputs.

2.3.2     Behavioral Requirements

The behavioral requirements define the actions taken when the software responds to internal and external stimulus. This includes describing what event causes an activity to start up and the event that causes it to stop.

2.3.3     Performance Requirements

The performance requirements specify the numeric requirements that are expected of the software as well as on the human interaction with the software. Examples of performance requirements may include:

�         Number of data transactions within a certain time frame

�         Number of simultaneous users

�         Number of terminals to be supported.

2.3.4     Operational Requirements

The operational requirements define the usability of the software as it communicates with its human users. This may include information concerning the qualifications of the users, level of detail for external messages, and Graphical User Interface standards.

Constraint

The constraint requirements define any restrictions that will impact the software that is being developed. Constraint requirements usually deal with:

�         Design / Development restrictions. Example: Target hardware and programming language.

�         Environment Restrictions. Example: Software size and resource utilization.

�         Data Restrictions. Example: Maximum number of files and size of files.

2.4         Software Requirements Engineering Overview

Software Requirements Engineering involves transforming system requirements into a description of software requirements, performance parameters, and software configuration through the use of an iterative process of definition, analysis, trade-off studies, and prototyping. There are many participants involved with analyzing and documenting the software�s requirements and their roles can be classified as:

2.4.1     Buyer

The buyers are the individuals that are responsible for the acceptance of the software. A buyer can be an internal department of an organization, external customer, or the general public. Whatever the origins, the buyers are responsible for contracting or paying for the software system.

2.4.2     End-User

The end-users are the individuals who ultimately use the system to perform some predefined set of tasks. Ultimately, the end-users will be the individuals who install, operate, and use the software.

2.4.3     Application Experts

The application experts are the individuals who provide domain knowledge and supply a more detailed understanding of the problem.

Software engineers.

The software engineers are the individuals who provide expertise on software design, prototype development, and technical feasibility. The software engineers works closely with the end-users, buyers, and application experts.

2.4.4     Requirements Engineer

The Requirements Engineers are the individuals who are responsible for the identification and documentation of the requirements. These individuals are also the mediate between the buyers, end-users, application experts, and the software engineers.

2.5         Software Requirement Engineering Activities:

The fundamental Software Requirements Engineering Activities that the requirement engineer performs are:

2.5.1     Context Analysis

The context analysis activity determines the underlying need of why the software is to be created. The technical, operational, and economic boundaries are determined and the scope of the software system is documented.

2.5.2     Requirements Elicitation

The requirement elicitation activity creates the required knowledge structure by gathering requirement-related information. There are many different types of methodologies that can be used to elicit information like:

�         Joint Application Development

�         Questionnaires

�         Interviews

�         Observation of operational environments

2.5.3     Requirements Assessment

The requirement assessment activity performs a risk assessment on the documented software requirements. The goal of this activity is to achieve an acceptable level of risk concerning the potential problems of:

�         Inconsistencies with the software requirements.

�         Ambiguities

�         Incompleteness

Exploration of design Feasibility

The exploration of design feasibility activity is concerned with assessing technical feasibility and making requirements tradeoffs may require a demonstration that a feasible software design does exist.

All requirements that are identified during the Requirement Engineering activities are documented in the Software Requirements Specification (SRS). The goal of the SRS is to describe all externally observed behaviors and characteristics expected of the software system.

3.   Software Requirements Engineering Concepts

In the Software Engineering community, there exist many examples of inconsistent terminology. These discrepancies must be resolved in order to advance the discipline and communicate the lessons learned from previous experiences.

3.1 Software Requirements Engineering

Software Requirements Engineering is defined as all activities devoted to identification of user requirements, analysis of the requirements to drive additionally requirements, documentation of the requirements as a specification, and validation of the documented requirements against user needs, as well as processes that support these activities.

3.2 Software Requirements Elicitation

Software Requirements Elicitation is any activity that has as its objectives to gather, determine, extract, or expose software requirements. Examples of Software requirement elicitation activities are:

Interviewing

System specification analysis

Brainstorming

Reuse of requirements

Role Playing

Prototyping

Analogy formulation

Requirements communication

Scenario generation

Creative exploration

3.3 Software Requirements Analysis

Software Requirements Analysis is any activity that has as its objective to organize, interpret, understand, classify, or assess feasibility, completeness, or consistency of software requirements. Examples of Software Requirements Analysis activities are:

Context analysis

Analysis of the SRS

Problem modelling

Performance modelling

Prototyping

Requirements feasibility assessment

Architectural exploration

Requirements classification

Analysis of requirements

Requirements prioritization

3.4 Software Requirements Specification

Software Requirements Specification is any activity that has as its objective to capture a description of the software requirements - usually the final form of this description is a software requirements specification (SRS). Examples of Software Requirements Specification activities are:

Requirements representation

Requirements Communication

Requirements Consistency checking

Requirements Completeness analysis

Prototyping

Writing the SRS

3.5 Software Requirements Validation

Software Requirements Validation is any activity that has as its objective to obtain buyer approval of the specification of the software requirements. Examples of Software Requirements Validation activities are:

Requirements representation

Requirements communication

Discussing

Interviewing

Customer Approval

Formal inspection

4.   Introduction to Requirements Elicitation

4.1     Outcome of a Good Elicitation Process - Users Perspective

After the Requirements Elicitation Process is completed, the users will have a better understanding of their needs and constraints. From this, the users will be in a position to evaluate alternatives and understand the implications of their decisions. This level of understanding is extremely important since it is usually the users who drive the requirements, which in turn drives the design and implementation of the entire software system. Thus, the quality of the requirements document is related to the users understanding of the issues and tradeoffs involved.

Another outcome of a good elicitation process is that the users and the developers form a common vision of the problem and the conceptualized software solution. It is the goal of the elicitation process to strive for this common understanding of all individuals involved in the software engineering process.

4.2     Outcome of a Good Elicitation Process - Developers Perspective

The process of elicitation is of vital importance to the developers of the software. This is the fundamental process that will be used to construct a clear high level specification of the problem that is to be solved. Other outcomes of a good elicitation process are developers who:

�         Are developing a solution to the right problem

�         Are solving a problem that is feasible from all perspectives

�         Have a high level specification of the solution

�         Have cooperative users

�         Have all the necessary information, explanations, and justifications

�         Can make proper design justifications

�         Need minimal ongoing interactions with the users

4.3     Underlying Difficulties with Elicitation

The process of elicitation of requirements is a difficult process. This comes from the fact that no single person has the complete picture. The only way to understand all perspectives involved is to work effectively as a coherent group. The following are some common difficulties associated with this process:

Articulation Problems

Articulation problems can occur because the users are usually experts in their application domain but not in the process of engineering software. Additionally, the engineers are experts in development and not in the users application domain. The users and developers having different vocabulary, terms, and concepts magnify this problem. Other factors that affect the Articulation problem include:

�         Users who are not aware of their needs

�         Users who are aware of their needs but are unable to articulate them

�         Users who are unwilling to prioritize and make tradeoffs

�         Users who misunderstand the issues

�         Developers who are not skilled in listening

�         Developers who fail to understand the users

�         Developers who are dominating in their approach to elicitation

�         Developers who are solution not problem orientated

Problem Complexity

The complexity of modern software system makes the process of requirements elicitation extremely difficult. These systems have interconnections between subsystems and environments that even expert in specialized disciplines have difficulty understanding.

Nature of Requirements

The most problematic aspect of any requirements elicitation process is the fundamental characteristic of the requirements engineering process:

4.4     Requirements change and migrate

�         Users learn and grow.

�         Requirements are diverse and conflicting

�         Multiple views can be difficult to integrate

�         Requirements can be difficult to evaluate

5.   Paradigms and Techniques for Enhancing Requirements Elicitation

There exists a set of methodologies, tools, and techniques that can be used to enhance the process of requirement elicitation. These high level paradigms emphasize selected themes that can be used to identify relevant sources of requirements, select appropriate questions, and aid in analyzing and gathering the required information. Examples of high level paradigms include:

�         Joint Application Development

�         Adaptive Loops Framework.

�         Prototyping

�         Contextual Inquiry

�         Critical Success Factors Analysis

Joint Application Development is a process methodology for promoting cooperation, understanding, and teamwork among buyers, users, and developers. It is a top-down methodology that consists of well-defined phases, tasks, and outputs which emphasize idea generation and evaluation. This is facilitated by participating in group requirements gathering workshops that are structured to bring together users and developers to identify and document user requirements.

5.1     Case study of Advance Information Services Inc.

Advanced Information Services Inc (AIS) has successfully used the JAD process and has incorporated it into its Organizational Standard Software Process. The organizations requirements phases stipulate:

�         Requirements gathering, analyses, and documentation occur within a JAD workshop.

�         Draft requirements are written before the JAD workshop and represent the software teams "first attempt" at documenting the requirement.

�         The draft requirements are then displayed and explained to the customer in the JAD session.

�         A discussion concerning the correctness of the draft requirement is undertaken with the customer and the software team.

�         Any updates to the draft requirements are made in the JAD workshop.

�         Any additional requirements that are identified are documented and will become the draft requirements in subsequent JAD workshops.

6.   Recognizing Resistance in Communication

6.1     What is Resistance?

In the requirement elicitation process, basic communication skills are very important for successful regardless of the method used for elicitation. Examples of such basic communication skills are:

�         listening to others.

�         Saying "no" without guilt or fear.

�         Allowing others to feel in control.

�         Asking open-ended questions.

�         Being authentic

Resistance is a physical expression of an emotional process-taking place within a person and takes the form of opposition. This is probably the most difficult aspect of communicating with other individuals since resistance to new ideas is a normal part of learning. Having the ability to identify resistance is useful to the requirement engineer since it signifies that an important issue has just been uncovered. The first step in identifying resistance is to define the different types of resistance. Examples of different types of resistance:

Types of Resistance

Examples

Overload Resistance

In a communication activity, the person asks for more and more information and no matter how information is given, it is never enough.

Flood of detail Resistance

The person keeps giving you so much detail that you understand less and less

Time Resistance

The person says he really wants to go ahead with the activity but the timing is off: he never has time to meet with you, or interrupts your meeting by taking phone calls, calling in the secretary, or talking with people who just "drop by".

Impracticality Resistance

The person keeps reminding you that he lives in the real world.

Not Surprised Resistance

No matter what bizarre and unexpected things happen in a project, the person claims not to be surprised.

Attach Resistance

The person attacks with angry words, a red face, pounding on the desk, pointing a finger in your face, and punctuating the end of every sentence.

Confusing Resistance

The person continuously claims to be confused, even after you have explained things several times.

Silence Resistance

The person does not react or respond, even when you push hard for him to express concurrence or objections.

Intellectualizing Resistance

The person wants to discuss theory after theory about why things are the way that they are.

Moralizing Resistance

The person puts himself on a pedestal by using phrases of superiority, instead of honestly confronting conflict of views

Compliance Resistance

The person always agrees with you and eagerly wants to know what�s next. Reservations are never expressed and the implications are that whatever you do is fine.

Methodology Resistance

The person asks endless questions about your methods in conducting the activities and you are grilled on petty details.

6.2     Handling Resistance

After resistance is identified, the following 4 steps can help handle and mitigate the resistance:

�         Identify the resistance that is taking place.

�         State in a non-accusing way the form of resistance that is taking place

�         Give a one sentence authentic reaction to the resistance and its effect on you.

�         Listen and let the person respond.

7.   The Software Requirements Specification

All requirements that are identified during Requirement Engineering are documented in the Software Requirements Specification (SRS). The goal of the SRS is to describe all externally observed behaviors and characteristics expected of the software system. The contents of a SRS include specifications for the inputs, outputs, system behaviors, and externally observable characteristics.

The SRS plays an important role in the developmental life cycle. It is the requirements that drive the design and implementation of the software, verification and validation, and training. Thus, any undesirable qualities in the SRS will be manifested in the software product.

In order to produce high quality software, the Software Requirements Specification should possess the following characteristics:

Characteristic

Description

Correct

�         Every requirement stated in the SRS is one that the software shall meet

Unambiguous

�         Every requirements stated in the SRS only has 1 logical interpretation

Complete

�         The SRS includes all significant requirements

�         The SRS includes all realizable classes of input data

�         The SRS includes full labels and references to all figures, tables and diagrams.

Consistent

�         No subset of individual requirements stated in the SRS conflict with other individual requirements

Verifiable

�         For every requirement stated in the SRS, there exists some finite cost-effective process with which a person or machine can check that the software meets the requirements

Modifiable

�         The entire SRS has a style and structure such that any changes to the requirements can be made easily, completely, and consistently which retains the structure and style.

Traceable

�         For every requirement stated in the SRS, it is clear of the requirements origin and it is possible to reference each requirement in future developments.

A SRS that possess these qualities can provide several benefits such as:

1. The SRS can establish the bases for contractual agreement between the customer and developers.

2. The SRS can establish the bases for performing cost, schedule, and resource estimates for the developmental effort.

3. The SRS can provide a baseline for verification and validation of the software.
