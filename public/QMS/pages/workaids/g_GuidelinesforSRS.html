1 Overview and Objectives

The Software Requirement Specification (SRS) document describes the functional aspects of the system. In order to proceed with the software development, the requirements stated in the proposal or the URS document need to be analyzed in detail prior to its translation into software design document (SDD).

The main objectives of the SRS document are:

�         To define the functionality required of the system/subsystem in detail.

�         To define the user interface, if a prototype is not prepared

�         To define the security mechanism

�         To form the basis for the design of the system

�         To form the basis for the System Test Plan

The SRS should be reviewed and signed by the client prior to the commencement of the subsequent phases of development.

3 Document Contents

3.1       Introduction

3.1.1 Background

This section gives a brief description about the client and the project.

3.1.2     Purpose of the Document

�         To define the functionality of the system/subsystem in detail.

�         To define the user interfaces, if a prototype is not prepared.

�         To define the security mechanism.

�         To form the basis for the design of the system.

�         To form the basis for the System Test Plan.

3.1.3     References

List the documents/systems referenced in creating this document such as the Proposal, Requirement Specifications etc.

3.1.4             Structure of the Document

Lists all the sections included in this document.

3.2       Summary of Requirements

Note: This section including the subsections is optional, if the requirements are specified in RS document.

3.2.1 Situation Analysis

This section specifies the current or foreseen problems with the existing system /sub-system or problems that could occur due to the non-existence of a system/sub-system.

3.2.2             Statement of Requirement

This section states the user requirements or enhancements of the current system to be implemented.

3.3       Project Description

Note: This section including the subsections is optional, if the description is covered in RS document.

3.3.1 In Scope

This section outlines the scope of the project.

3.3.2             Out of Scope

This section describes clearly what is beyond the scope of the project.

3.3.3             Customer Perspective

This section contains the requirements for the system from the client's business perspective. This typically includes increased ability to change, elimination of duplication of work, increase productivity etc.

3.4 Process Model

3.4.1 Process Decomposition

Each process or module is specified diagrammatically.

3.4.2 Process Description

Each screen or page is described with its objective, inputs, outputs and functionality. Screen layouts should be documented.

3.4.3 Data Flow Diagram

Describes how the data flows between various processes.

3.4       Data Model

Note: This section is optional i.e., it could be covered in the High Level Design document.

3.4.1 Entity Description

This section describes all the entities along with their attributes.

3.4.2 Entity Relationship Model

In this section the relationship between different entities of the product is shown diagrammatically.

3.5       External Interfaces

This section lists all the external interfaces for the system/subsystem.

3.6       Security

This section lists different security levels of the system. Describes how the system is controlled and security is implemented.

3.7       User Interfaces

This section specifies how the user interaction will be provided.

4 Non-Functional Requirements

4.1 Reliability:

4.1.1 Maturity

Mean Time Between Failures (MTBF) = Total software system operation time

Number of failures

Note1: Failures could be classified as fatal, Major and Minor in which case different values of MTBF can be computed for the three types of failures.

 

4.1.2 Fault Tolerance:

Breakdown Ratio =                      Number of breakdowns

Number of failures

 

4.1.3 Recoverability:

Mean Recovery Time =                                   Total number of hours needed for recovery

           Number of breakdowns during observation period

4.2 Usability

Understandability = Time taken to understand the logic of the software

Learn ability = Time taken to learn to use the software

Operatabilty = Time taken to operate and control the software

Installation time = Total time required for installation

Message explicitness =

 Number of messages that explicitly describe causes and counter measures

                                  Total number of messages

4.3 Efficiency

Time Behavior

a) Response time = Time taken for one transaction

b) Throughput = Number of transactions per unit time

Resource Utilization

a) CPU Utilization =           Total CPU processing time

                 CPU operating time

b) Main Memory Utilization =           Amount of main memory used

                   Main memory capacity

4.4 Maintainability

Mean time to Repair (MTTR) =                 Total repair time

Number of failures

Note2: Failures could be classified as critical, semi-critical and non-critical in which case different values of MTTR can be computed for the three types of failures.

Analyzability

Mean time to Analyze (MTTA) =             Total analysis time

Number of failures

Changeability

Mean time to change (MTTC) = Total time to implement the change

Number of failures

Stability

Mean time to test for impact of changes (MTTI) =  Total time for regression testing

                                                                                       Number of failures

Testability

Mean time to test (MTTT) =       Total time for validation

                                               Number of validation

4.5 Portability

4.5.1 Adaptability

a) Hardware independence:

Machine applicability rate = Number of machine models that can run the target software

           Number of available machine models  

b) Software independence:

OS applicability rate = Number of operating systems that can run the target software

        Number of available operating systems

c) Data independence:

Data applicability rate =          Amount of data used by the ported software

          Amount of data available

4.5.2 Install ability

Mean Installation Time =         Total time required for porting work

           Number of porting functions

 

4.5.3 Replace ability

a) Source Code Revision Rate =              Number of lines of source code requiring revision

                      Total Number of lines in porting software

b) Database Revision Rate =                Number of lines of source code requiring revision

                Total Number of lines in porting software

4.6 Functionality

4.6.1.1 Suitability:

Ratio of user requests for improvements = Number of user requests for improvements

                                                                               Time elapsed since product shipment

Requirement specification =                       Number of function corrections in requirement specification

                                                         correction ratio Number of product function in requirement specification

4.6.2 Accuracy

Accuracy =           Degree of precision provided

         Degree of precision required

4.6.3 Interoperability

Interoperability =        Number of interface / protocols that agree

                                    Total number of interface / protocols

4.6.4 Security

Encryption Ratio =          Amount of encrypted data

                                       Amount of encrypt able data

a) Access History Ratio =     Amount of data with access history

                                                       Total amount of data

b) Access History Ratio =      Number of times access violations occurred

                                                    Total number of times accessed
