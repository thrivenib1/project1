Guidelines for Testing Requirement Specification

GLVAR02

Version: 3.00

Date: 17th May 2010

 

1 Overview and Objectives

The objective of these guidelines is to help in the preparation of Testing Requirement Specification for a Testing Project. Testing is the process of exercising or evaluating a system or system component by manual or automated means to verify that it satisfies specified requirements.

The objective of Testing Requirement Specification document is

·         To define the scope of the testing project

·         To specify effort, cost and schedule

·         To define the testing approach

·         To define the acceptance mechanism

Testing Requirement Specification details may be included in Project Proposal or Project Plan, in which case, this document may not be prepared separately.

3 Document Contents

3.1 Introduction

3.1.1 Background

This section mentions a brief summary of the client’s business and a brief background of the project.

3.1.2     Purpose of the Document

The purpose of the Test Requirement Specification is

·         To understand and define the scope of the testing project

·         To specify effort, cost and schedule

·         To understand the testing approach

·         To define the acceptance mechanism.

3.1.3     Reference

All documents or systems referenced in creating this document should be mentioned in this section.

3.1.4     Structure of the Document

This section lists all the sections included in this document.

3.2 Summary of Requirements

3.2.1 Testing Requirements

The testing requirements in terms of usability, performance, stress, functionality, security, portability and multi-user usability etc. should be mentioned in this section.

3.2.2 In Scope

The scope of the testing e.g. products to be tested is specified in this section.

3.2.3 Out of Scope

Activities that are beyond the scope of the project (e.g. to improve performance help desk) are mentioned clearly in this section.

3.2.4 Assumption and Dependencies

Assumptions made and dependencies, if any, are specified in this section.

3.2.5 Test Environment

The environment in which the software will be tested is specified in this section.

3.3 Baseline

3.3.1 Documents

All the documents provided by the client e.g. Requirement Specification, Functional Specification, User Manual etc. are listed in this section.

3.3.2 Initial Status of the Product

How the accuracy (absence of defects), completeness (full data and software) and currency (most recent version) of the baselined software will be established is specified in this section.

3.4 Execution Method

3.4.1 Testing Project Life Cycle Model

The Testing Life Cycle Model followed for the project execution e.g. Product QA Life Cycle Model in Process Manual (QMS-02) should be specified in this section.

3.4.2 Documenting test results

Test records to be maintained for all the tests carried out on the system are specified in this section.

3.4.3 Bug fixing

The person (or party) responsible  (customer / developers / testers) for bug fixing is specified in this section. Also how the bug fixing will be done (as and when they come or at the end of testing) should be specified.

3.4.4 Configuration management

The change control procedure including check out / check in mechanism for the configurable items are specified in this section.

3.4.5 Cycles of testing

The numbers of test cycles performed to test the product are specified in this section.

3.5 Test deliverables

All the deliverables after the test e.g. test cases, test results, test metrics etc. are listed in this section.

3.6 Acceptance procedure

The acceptance criteria are specified in this section.

3.7 Metrics

Metrics to be collected are specified in this section.
