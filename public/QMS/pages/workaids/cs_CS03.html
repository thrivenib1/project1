Coding Standard for C++ Language
CS-03 Version: 3.00 Date: 17th May 2010

3.0 Introduction

3.1 Purpose

The purpose of this Standard is to provide coding standard for C++.

3.2 Scope

This Standard is applicable for developing C++ based projects.

3.3 Source Code in Files

The following conventions are suggested. The convention used should always be uniform across all the files.

�         Include files in C++ always have the file name extension ".hxx". The extension should be consistent across all include file names.

�         Implementation files in C++ always have the file name extension ".cxx".

�         Template definition files always have the file name extension ".txx".

�         Every file that contains source code must be documented with an introductory comment that provides information on the file name and its contents as shown below:

/******************************************************************

Copyright information : Imaje, Inc.

File name : imajproj.C

Project name : P-API

Date of creation : 09/26/2000

Version number : Ver. 1.0

Brief description : This file contains implementation of

Author name : Vinod

************************************************************/

When the code undergoes a modification, it should be preceded by documentation about the modification Summary as shown below:

/*****************************************************************

Modification Summary

Date:

Description:

Author name:

Version:

*****************************************************************/

Every class should be documented with introductory comments that provide information about the class as shown:

/****************************************************************************************************************

Class name : TopicTemplate_i

Class type : Concrete

Brief description : Implementation class for the TopicTemplate IDL

Super classes : TopicTemplateBOAImpl

Sub classes : None

Classes used : None

Author name : Vinod

****************************************************************************************************************/

Every method should be documented with introductory comments that provide information about the method as shown:

/***************************************************************************************************************************

Name : createChampion

Synopsis : gets the user details of a particular user.

Type : instance method

Input parameter : long topicId, const

 char * championId, environment IT_env

Output parameters : None

Return Type : CORBA::ShortGlobal variables : None

Calling Synta

x : createChampion(topicId, championId);

Author name : Vinod

***************************************************************************/

�         Every include file must contain a mechanism that prevents multiple inclusions of the file. The easiest way to avoid multiple includes of files is by using an #ifndef/#define block in the beginning of the file and an #endif at the end of the file as shown below:

#ifndef FOO_HXX

#define FOO_HXX

// The rest of the file

#endif

Never specify relative UNIX names in #include directives.

// NOT RECOMMENDED
#include <../include/fnutt.h>

�         Every implementation file is to include the relevant files that contain:

�         Declarations of types and functions used in the functions that are implemented in the file.

�         Declarations of variables and member functions used in the functions that are implemented in the file.

�         An include file should not contain more than one class definition.

�         Always give a file a name that is unique in as large a context as possible.

�         An include file for a class should have a file name of the form <class name> + extension. Use uppercase and lowercase letters in the same way as in the source code.

�         Write some descriptive comments before every function.

�         Use // for single line comments.

�         Use the directive #include "filename.hxx" for user-prepared include files.

�         Use the directive #include <filename.hxx> for include files from libraries.

�         Every implementation file should declare a local constant string that describes the file so the UNIX command "what" can be used to obtain information on the file revision.

static const char* sccsid =
"@(#) Mail_i.C, rev. A, Copyright Imaje, Inc. 1996";

3.4         Naming Standards

�         In names, which consist of more than one word, the words are written together and each word that follows the first is begun with an uppercase letter (Hungarian Notation).

�         Do not use identifiers which begin with one or two underscores (`_' or `__').

�         Do not use typenames that differ only by the use of uppercase and lowercase letters.

�         Names should not include abbreviations that are not generally accepted.

�         Choose variable names that suggest the usage.

�         Variables local to a method will be named as

<data type specifier><meaningful name in hungarian notation>

Variables at class level will be named as

<underscore><data type specifier><meaningful name in hungarian notation

The data type specifiers could be

i � integer

b � Boolean

s � string

c � character

p � pointers etc.

�         Any Object instance should be prefixed by its class name. The class name however will start with a lowercase character in this case.

�         Methods should be named in the format: firstWordLowerCaseButInternalWordsCapitalized

�         The table below describes the rules for identifying constants, classes, and Exception classes.

Identifier

Naming Convention

constants

UPPER_CASE_WITH_UNDERSCORES

Classes

FirstWordCapitalizedWithInternalWordsAlsoCapitalized

<ClassName>Exception

3.5         Style

�         The public, protected, and private sections of a class are to be declared in that order (the public section is declared before the protected section, which is declared before the private section). List the member function prototypes in the header file in the same order they occur in the source file.

�         No member functions are to be defined within the class definition. See example below:

// Instead of writing like this:
class String
{
public:
int length() const // No !!
{
return _iLen;
}
// ...
private:
int _iLen;
};

// Do it this way:
class String
{
public:
int length() const;
// ...
private:
int _iLen;
};

inline
int
String::length() const
{
return _iLen;
}

�         Always create a default constructor for a new class, especially when the class contains pointers or references as data members. Don�t rely on the compiler to do what you want.

�         Include a short comment in the class header file that describes each member function.

�         Always provide the return type of a function explicitly.

�         When declaring functions, the leading parenthesis and the first argument (if any) are to be written on the same line as the function name. If space permits, other arguments and the closing parenthesis may also be written on the same line as the function name. Otherwise, each additional argument is to be written on a separate line (with the closing parenthesis directly after the last argument).

�         Right and wrong ways of declaring formal arguments for a function (in function definition) are shown below:

// Right:
int
myComplicatedFunction( unsigned uValue,
int iValue,
char* pCharValue,
int* pIntValue,
myClass* pmyClassValue,
unsigned* pUnsignedValue );

// Wrong:
int
myComplicatedFunction( unsigned uValue, int iValue,
char* pCharValue, int* pIntValue,
myClass* pmyClassValue, unsigned* pUnsignedValue );

�         In a function definition, the return type of the function should be written on a separate line directly above the function name as in the previous example.

�         Always write the left parenthesis directly after a function name as shown.

void foo (); // No!!
void foo(); // Better

�         Braces ("{}") which enclose a block are to be placed in the same column, on separate lines directly before and after the block.

�         The flow control primitives if, else, while, for and do should be followed by a block, even if it is an empty block.

�         The de-reference operator `*' and the address-of operator `&' should be directly connected with the type names in declarations and definitions.

�         Do not use spaces around `.' or `->', nor between unary operators and operands.

3.6         Classes

�         Avoid public or protected member data where private data will suffice.

�         Access functions are to be inline.

�         Forwarding functions are to be inline.

�         Constructors and destructors must not be inline. This is because we should always avoid calling inline functions from within inline functions.

�         Friends of a class should be used to provide additional functions that are best kept outside of the class.

�         A member function that does not affect the state of an object (its instance variables) is to be declared const.

�         If the behaviour of an object is dependent on data outside the object, this data is not to be modified by const member functions. (same as above)

�         A class which uses "new" to allocate instances managed by the class, i.e. instances bound to member variables of pointer or reference type that are de-allocated by the object, must define a copy constructor.

�         All classes, which are used as, base classes and which have virtual functions, must define a virtual destructor.

�         A class which uses "new" to allocate instances managed by the class, i.e. instances bound to member variables of pointer or reference type that are de-allocated by the object, must define an assignment operator.

�         Use operator overloading sparingly and in a uniform manner.

�         When two operators are opposites (such as == and !=), it is appropriate to define both.

�         A public member function must never return a non-const reference or pointer to member data.

�         A public member function must never return a non-const reference or pointer to data outside an object, unless the object shares the data with other objects.

�         A derived class often requires access to base class member data in order to create useful member functions. The advantage in using protected member functions is that the names of base class member data are not visible in the derived classes and thus may be changed. Such access functions should only return the values of member data (read-only access). This is best done by simply invoking const functions for the member data.

3.7         Functions

�         Do not use unspecified function arguments (ellipsis notation).

�         Always specify the return type of a function explicitly.

�         A public function must never return a reference or a pointer to a local variable.

�         Avoid the use of preprocessor directive #define to obtain more efficient code; instead, use inline functions where appropriate. E.g.:

// Example of problems with #define "functions"
#define SQUARE(x) ((x)*(x))
int a = 2;
int b = SQUARE(a++); // b = (2 * 3) = 6

// Inline functions are safer and easier to use than macros if you
// need an ordinary function that would have been unacceptable for
// efficiency reasons.
// They are also easier to convert to ordinary functions later on.
inline int square( int x )
{
return ( x * x );
};

int c = 2;
int d = square( c++ ); // d = ( 2 * 2 ) = 4

�         Try to avoid long and complex functions.

�         Try to avoid functions with many arguments.

�         If a function stores a pointer to an object, which is accessed via an argument let the argument have the type pointer. Use reference arguments in other cases.

�         When overloading functions, all variations should have the same semantics (be used for the same purpose).

�         The names of formal arguments to functions are to be specified and are to be the same both in the function declaration and in the function definition.

�         Minimize the number of temporary objects that are created as return values from functions or as arguments to functions.

3.8 Constants

Avoid the use of numeric values in code; use symbolic values instead.

3.9 Variables

�         Variables are to be declared with the smallest possible scope.

�         Each variable is to be declared in a separate declaration statement.

�         Every variable that is declared is to be given a value before it is used.

�         If possible, always use initialization instead of assignment

// Do not do this!
// int i;
// ... 1022 lines of code
// i = 10;

int j = 10; // Better

3.10      Pointers and References

�         Do not compare a pointer to NULL or assign NULL to a pointer; use 0 instead.

�         Pointers to pointers should be avoided whenever possible.

�         Use a typedef to simplify program syntax when declaring function pointers.

#include <math.h>

// Ordinary messy way of declaring pointers to functions:
// double ( *mathFunc ) ( double ) = sqrt;

// With a typedef, life is filled with happiness (Chinese proverb):
typedef double MathFuncType( double );
MathFuncType* mathFunc = sqrt;

void
main()
{
// You can invoke the function in an easy or complicated way
double returnValue1 = mathFunc( 23.0 ); // Easy way

// No! Correct, but complicated
double returnValue2 = ( *mathFunc )( 23.0 );
}

3.11      Type Conversions

�         Avoid using explicit type conversions (casts) on simple data types.

�         Avoid functions that use implicit type conversions.

�         Never convert pointers to objects of a derived class to pointers to objects of a virtual base class.

3.12      Flow Control Structures

�         The code following a case label must always be terminated by a break statement.

�         A switch statement must always contain a default branch, which handles unexpected cases.

�         Never use goto.

�         The choice of loop construct (for, while or do-while) should depend on the specific use of the loop.

�         Always use unsigned for variables, which cannot reasonably have, negative values.

�         Always use inclusive lower limits and exclusive upper limits.

�         Avoid the use of continue.

�         Use break to exit a loop if this avoids the use of flags.

�         Do not write logical expressions of the type if(test) or if(!test) when test is a pointer.

3.13 Expressions

Use parentheses to clarify the order of evaluation for operators in expressions.

3.13      Memory Allocation

�         Do not use malloc, realloc or free. Use new for allocating memory and delete for reallocation.

�         Always provide empty brackets ("[]") for delete when reallocating arrays.

�         Avoid global data usage.

�         Do not allocate memory and expect that someone else will reallocate it later.

�         Always assign a 0 value to a pointer that points to reallocated memory.

3.14      Fault Handling

�         Check the return codes, which may be received from library functions even if these functions seem foolproof.

3.15      Portable Code

�         Do not assume that an int and a long have the same size.

�         Do not assume that an int is 32 bits long (it may be only 16 bits long).

�         Do not assume that a char is signed or unsigned.

�         Always set char to unsigned if 8-bit ASCII is used.

�         Be careful not to make type conversions from a "shorter" type to a "longer" one.

�         Do not assume that pointers and integers have the same size.

�         Use explicit type conversions for arithmetic using signed and unsigned values.

�         Do not assume that you know how an instance of a data type is represented in memory.

�         Do not assume that longs, floats, doubles or long doubles may begin at arbitrary addresses.

�         Do not depend on underflow or overflow functioning in any special way.

�         Do not assume that an object is initialized in any special order in constructors.

�         Do not assume that static objects are initialized in any special order.

�         Do not write code, which is dependent on the lifetime of a temporary object.Avoid using shift operations instead of arithmetic operations.

�         Avoid pointer arithmetic.
