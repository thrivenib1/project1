Coding Standard for Shell Scripts
CS-01 Version: 3.00 Date: 17th May 2010

1.0 Introduction

1.1     Purpose

The purpose of the Standards is to provide the coding standard for UNIX Shell Scripts.

1.2     Scope

This standard is applicable for developing UNIX Shell Scripts.

1.3     When to use a shell procedure

�         Doing the same task repeatedly.

�         A complex program using many command lines

1.4     When not to use a shell procedure

When the requirement is too complex for the shell procedure to achieve.

�         Requires high efficiency.

�         Requires a different Hardware environment.

�         Requires different software tools.

1.5     Standards for shell procedure Designing

�         A shell program should be based on a logical division of functions.

�         For each logical division, create a separate function or program that will work to accomplish that task. These shell programs and functions can call other functions and programs as needed.

�         A shell procedure should have the following features.

�         Reliability

�         Efficiency

�         Portability

�         Usability

�         In any shell program simplicity should prevail over complexity.

1.6     How to write a shell procedure

Try to make the script a generic one by passing the variables from the command line.

1.7     Make a shell procedure structured

�         Divide the program in to manageable chunks to derive the following advantages

�         Makes the program easier to understand.

�         Reduces redundant code (calls the function instead of repeating the code many times).

�         Makes debugging easier.

�         Increases maintainability.

�         Encourages reusing code.

�         Decreases testing time

�         Don't define a function with a name, which is aliased.

�         Define all the functions in one file and execute the same in the program where you want to use the functions or assign the path the filename, which contain all the functions to FPATH and export it.

�         Use return statement in functions when you want to come out the function rather than exit which will exit out from the program.

�         When functions are used in a shell program, care should be taken to store all the arguments passed to the shell program because when a function is called with arguments inside a shell program, the function arguments will replace the originally passed arguments to the shell program.

�         Don't define a function using a name that already exists, one of the two things will happen. If the name is a function you will redefine the function. If the name is an alias you will not redefine the alias but will instead run the alias.

�         Use "typeset" command to define variables in a function, if they are not used in the main program.

�         Any time that if we find copying or duplicating code, we should consider utilizing a function. Some things that often end up being function in many of the shell programs are :

�         Error routines - No need to produce the same codes for every error; make it in a function.

�         Input/output routines - Often we read/write to/from the terminal or a file in several places. No need to copy the same codes; make it a function.

�         Any repeated process - Any procedure that repeat more than 2 times should be a function

�         When we are writing a new function or program to accomplish some task there are 2 question's to ask ourselves:

�         Is there some way that we could generalize this function or procedure?

�         Is there a way to make the module independent from the current calling program so that other programs can share the code?

�         When you are repeatedly performing the statements in the function call by calling the function multiple times, go for recursive functions.

1.8     Standards for shell procedures

�         Always use the Bold Brackets compared to test command.

�         Use curly braces when retrieving the value of the variable.

�         Re direct unnecessary errors messages to /dev/null

�         Never take the return status from the pipe, as the status returned from the pipe command only tells about the how the last command executed but not the status of the whole pipe command.

�         Use case statements when many decisions are made rather than if-else loop.

�         It is always recommended to use $@ to $*.

�         Use exit/return commands with proper return codes, as these return codes have some meaning to operating system. (Refer to Unix manual for return codes).

�         Use ${parameter:-, $parameter:=; $parameter:?, $parameter:+ rather than going in to a if else loop, to set or unset a parameter.

�         A shell programmer can improve the efficiency by running large complex programs during non-prime time by using "at" command.

�         Shell-procedure efficiency depends on the number of bins searched for commands and their ordering is often prime candidates for efficiency improvement. Reduce the length of the value of PATH variable to the minimum possible.

�         Use to change in to a directory rather than use a long path name repeatedly.

�         Reduce the number of temporary files used in a shell program. Pipes and better selection of commands can reduce the number of temporary files.

�         There are many Unix commands where one can be substituted for several others. The usage of a particular command to reduce complexity is left to programmer discretion.

�         Try to use the built-in commands of shell instead of called programs (like awk etc ).

�         Put reducing filters (example grep, cut, awk ) in a pipe first to reduce the transfer of data from one command to the other in a pipe.

�         To reduce portability of shell programs from one machine to other, stick to the standard Unix commands contained in /bin and /usr/bin

�         Take the value of $? (Return code of some command) to some variable before it is changed by some other command.

�         Variable names should derive the use of variables for easy understanding and maintenance of any procedure.

�         The choice between the "for" and "while" loops is left to the programmer style, but the choice should affect the simplicity of the program.

1.9     Comments

�         Use comment wherever required. Comments should be meaningful and sufficient not cryptic or verbose.

�         Comments are required for each data item, logical data group and logical block of the program code.

�         All comments must precede the actual program code described, except for the line comments on the same line as that of the code they describe.

�         Comments should be placed at the same indentation level as that of the code they describe.

�         Code, which is machine dependent, should be commented to facilitate its portability.

�         Judicious use of comments is recommended as it helps to make shell programs easier to debug and to maintain

1.10  Testing/Debugging

�         Script need to be tested before it is delivered.

�         The options shell provides for debugging are -v and -x options

�         -v option: Shell will display every command line as it is written in the shell procedure

�         -x option: Shell will display each command executed and the values substituted for variables.

�         Use a global variable to turn debugging ON and OFF in shell procedures

�         There are 2 major types of error conditions that you should check in your shell script if it to be well behaved and of high quality.

�         Internal errors in the script

�         External errors

�         Internal errors are caused by some command failing to perform as expected or the user is not providing the necessary information to the shell program.

�         External errors are sent to shell procedure in the form of a signal sent by the system telling that some system-level event has occurred that will have an effect on your shell procedure.

1.11  How to Handle Internal Errors

�         In general a programmer needs to insure that the user is not allowed to provide invalid input that will cause the shell to behave independently

�         An integral part of well designed user interface is the quality of error messages provided as well as the degree to which the shell procedure protects the user from erroneous input that will cause the problems in the procedure.

�         Each time a shell command is executed the special variable $? is set to the return status of the command which provides you with the ability to check and ensure that the command worked properly.

1.12  How to Handle External Errors

Use TRAP command to handle the external errors (for example signals). TRAP can handle increasingly complex jobs as required.
