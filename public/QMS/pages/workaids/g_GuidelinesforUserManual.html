User Manual Guidelines

GLDES03

Version: 3.00

Date: 17th May-2010

 

1 Overview and Objectives

A user manual is designed to provide users with a good working knowledge of the Software Application/Product and to enable them to navigate smoothly through its various functions.

The purpose of these guidelines is to define broadly the structure of a user manual. 

These guidelines are applicable to all user manuals unless the customer particularly likes to have any modifications in the structure or contents.

2 Document Structure

A user manual should have the following structure.  

1. Cover Page

2. Copyright Information

3. About the Manual

3.1 Target Audience

3.2 Organization of the manual

3.3 Conventions used in the manual

4. Table of Contents

5. Chapter 1: Introduction

5.1 Overview of the software

5.2 Working environment

5.3 Installation of the software

5.4. Running of the software

6. Chapter 2: Quick Start Guide (optional)

7. Chapter 3: Chapter n

8. Appendices

8.1 Error Messages

8.2 Table of Figures

9. Glossary (optional)

10. Troubleshooting Guide

11. Index

12. Revision History

3.4         Table of Contents

Should be preferably auto-generated. Should include all the levels of sub-sections.

3.5 Chapter 1: Introduction

3.5.1 Introduction to the Software

Introduction, as the name suggests, should acquaint the end user with the basic purpose of the software. It should list all the important features of the software and preferably include a comparison with the currently available software developed for the same purpose.

3.5.2 Working Environment

The hardware and software environment required for running the software should be clearly specified.  Preferably, both the minimum and optimum values should be mentioned.

3.5.3 Installation

The installation procedure should describe the steps involved in installing the software.  It should clearly specify  any prerequisite software running on the system, any hardware expected to be connected or installed on the system for a better performance, or any order to be followed if different modules are installed separately.

3.5.4 Running of the Software

Method to access the software easily is described. 

3.5         Quick Start Guide

Basic working of the software is described in a few steps.  References can be given wherever a better understanding of the software is required.

3.6         Chapter 3: Working of the Software

The user documentation should, ideally, be a steadily growing presentation, with the earlier part catering to the needs of a beginner and the latter part the more experienced user of the software. In effect, the first part could be tutorial material and the second half could be reference material.

A two-part manual with a reference section and a tutorial section could be good as it gives the reader a choice in the amount and nature of the information.

The best design for software documentation is the one that suits the user of the software, and that requires least attention and learning. This would be a self-effacing design and is task-oriented. Software-oriented documentation has a design that concentrates on the structure and utilities of a program and would help in accessing the utilities easily.

Menu-oriented documentation has a design based on the order of the menus and menu options in the user interface of the software. This is all right if the original program designers were task oriented. However, if the UI design is not task oriented, the reader might end up having to look somewhere at the end of the manual for the first step

User role-oriented documentation is divided to suit the needs of different users like operators, application programmers and system administrators. However, this cannot be effective as the user roles are not always precisely defined, there are too many roles and some roles cover several distinct tasks requiring different types of information.

The task oriented documentation focuses on information needed to perform each specific task. The tasks are identified by analyzing the software and not by the type of user. It also helps in deciding exactly how much and what information is needed to go into the manual.

For developing user documentation including online help, refer to Guidelines on Development of User Documentation (GL-18).

3.7         Appendices

Provide Error messages, Table of figures and  any other information required for efficient and optimal use of the software.

3.8         Glossary

Provide a glossary specific to the software as well as important generic terms.

3.9         Troubleshooting Guide

Provide a troubleshooting guide for the software.   This can also mention the limitations of the software.  Refer to the Assumptions and Dependencies section of the project plan.

3.10      Index

Provide index for the user manual.

3.11      Revision History / Amendment Record

Revision History is mandatory for every user manual.  Maintain a clear document of all the changes made to the document, including the reason for change and who requested the change etc.
