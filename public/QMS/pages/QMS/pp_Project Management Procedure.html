17.0 Introduction

17.1.1 Purpose
The purpose of Project Management is to establish and maintain project plans that define project activities, track project progress, take corrective action when the performance deviates from the target, integrate relevant stakeholders according to an integrated process, form and sustain an integrated team for the development of work products.

17.1.2 Policy (Plan the Process, and Establish a Defined Process (GP2.1, GP 2.2 and GP3.1))

The organization is committed to using the best practices of Project Management to achieve the organization’s strategic plan. Project Management processes include initiating, planning, executing, monitoring, controlling and closing project activities to successfully satisfy Customer requirements.

The organization expects that every project establish and use a project defined process from the start of the project to the close of the project. The project’s defined process is used to manage the project and collaborate with Stakeholders. Every project will create a project plan to record the activities associated with the project defined process.

Integrated Project Management will be used to coordinate and collaborate with stakeholders throughout the life of the product or deliverable to better satisfy Customer needs, expectations and requirements.

The performance of the project will be monitored against the project plan and corrective actions are required when the performance results deviate from the plan. Project monitoring and controlling activities will be described in the project plan.

The specific practice of “Establish and Maintain Project’s Defined Process” in Integrated Project Management (IPM) will support and implement the generic practice “Establish a Defined Process).

17.1.3 Scope
The project management procedure encompasses the following CMMI process areas: Project Planning, Project Monitoring and Control, Integrated Project Management for Integrated Teaming

17.1.4 Entry criteria
Project management commences as soon as the contract is signed. Project management activities follow after Requirements Development activities.

17.1.5 Inputs

  1. RFP, URS, SOW
  2. Contract, PO,

Contract, Purchase Order, Statement of Work, User Requirement Specification, Estimates, RFP, and Proposal (technical and commercial) are the required inputs to start the project management activities. In some projects, all of these input documents may not be available. The missing inputs should be mentioned in the project kick-off meeting for necessary action.

17.3.2.1 Plan stakeholder involvement (PP/SP2.6)

Plan the involvement of identified stakeholders.

Process Steps:

  1. List relevant stakeholders in the project plan.
  2. Specify the roles and responsibilities of the relevant stakeholders in the project for each project life-cycle phase.
  3. Record the activities of the different stakeholders and inter-dependencies in the WBS.

17.3.2.3 Identify Project Risks (PP/SP2.2)
Identify and analyze project risks.

Process Steps:

  1. Specify the dates on which risk planning will take place in the project schedule.
  2. Identify and list risks associated with configuration management, quality management, cost management, supplier management and other support functions.
  3. Analyze the risks to determine the impact, probability of occurrence, and time frame in which problems are likely to occur. Prioritize the risks. Document the identified risks, impact, probability, time frame and prioritization of risks.
  4. Review and obtain agreement with relevant stakeholders on the completeness and correctness of the documented risks.

17.3.2.4 Estimate Scope of the Project (PP/SP1.1)

Establish a top-level work breakdown structure (WBS) to estimate the scope of the project

Process Steps:

  1. After project kick-off, top level WBS is prepared and entered in PCC.
  2. Prepare detailed WBS using the estimation sheet prepared after completion of SRS.
  3. Record the tasks, descriptions, dependencies and work products in the WBS.

17.3.2.5 Define Project Life Cycle (PP/SP1.3), Project Defined Process (IPM/SP1.1) and Plan for Project Work Environment (IPM/SP1.3)

Define the project life-cycle phases upon which to scope the planning effort. The project phases for software typically include selection and refinement of a software development model to address interdependencies and appropriate sequencing of software project activities.

Establish and maintain the project's defined process from project start-up through the life of the project. The project’s defined process consists of processes that form an integrated and coherent lifecycle for the project.

Establish and maintain the project work environment based on the organization’s work environment standards

Process Steps:

  1. Integra has developed lifecycle models called i-FI Model (Integra Fractal Integrated Model), testing model and i-LANG model for software development, testing and localization projects respectively. Other standards SDLC models like Agile, Spiral have been customised to suite the need to the organisation. Exploratory model has been developed for projects with new technology involving R&D. The process models are available in the process model section of the QMS.
  2. Select an appropriate project lifecycle model from the list of process models available in the QMS.
  3. Record the lifecycle model selected for the project in the project plan.
  4. Tailor the lifecycle model according to tailoring guidelines to meet a specific project needs
  5. In case the available lifecycle model is inadequate to meet the specific needs of the project, then select the project process from the organization set of standard processes that fits the needs of the project.
  6. Tailor the organization set of standard processes to create the project defined process
  7. Conduct SEPG meeting so that the tailoring and deviations to the organization standard process is approved.
  8. The Project Manager will use Project Control Centre to enter WBS, track issues, record minutes of meeting, reports, and infrastructure request for IT requests, among other features.
  9. The Project Manager will use SVN for Configuration Management. Wherever appropriate, use alternate tools such as applicable for the specific project.
  10. The Project Manager will take the support of Information Technology Department and Admin Department in supporting the project in the areas of maintenance and troubleshooting of Hardware, Software, Networking, Access Control, Configuration Management Systems, and interaction with Vendors. The requests to IT department and Admin department are logged and tracked to closure. Pending issues are reported in the weekly meetings and resolved. Urgent action items are informed to the BU head for resolution.

17.3.2.6 Plan Project Resources (PP/SP2.4) and Identify Knowledge and Skills (PP/SP2.5)

Plan for necessary resources to perform the project. Plan for knowledge and skills needed to perform the project. Plan for knowledge and skills needed to perform the project.

Process Steps:

  1. The initial effort, cost and schedule estimates along with initial WBS prepared during proposal phase is used to define the project resources.
  2. Expand initial or high level WBS to create a detailed WBS. Decompose into work packages that can be separately assigned, performed and tracked.
  3. The PM will request for infrastructure as per Infrastructure Request and Release.
  4. All customer-supplied material will be handled as per procedure for Customer Supplied Material.
  5. At project closure, Infrastructure is released as per Infrastructure Request and Release. Customer supplied material is returned as per the Control of Customer Supplied Material.
  6. Identify the knowledge and skills needed to perform the project and document in the project plan.
  7. Refer to the respective PA document to identify skills required by the processes followed by the project.
  8. Refer to the HR manual for specific skills required for specific jobs and roles. This information is scheduled to be made part of the QMS.
  9. Assess the knowledge and skills available and document in the project plan
  10. Raise a training request to the training department to provide needed knowledge and skills.

17.3.2.7 Use Organizational Process Assets to Plan Project Activities (IPM/SP1.2), Establish Estimates of Work Product and Task Attributes (PP/SP1.2) and Determine Estimates of Effort and Cost (PP/SP1.4)

Use the organizational process assets and measurement repository for estimating and planning the project activities. Establish and maintain estimates of the attributes of the work products and tasks. Estimate the project effort and cost for the work products and tasks based on estimation rationale.

Process Steps:

  1. Refer to the organization measurement repository to compare the current project with past projects to identify differences and similarities. Project planning parameters include application domain, and experience of the staff.
  2. Historical data include size of work product, work product attributes, effort, cost, schedule, staffing, defects, response time, supplier performance, service capacity, etc. Record the assumptions and rationale used to select the historical data.
  3. Size estimation
  4. Effort estimation
  5. Cost estimation
  6. Schedule
  7. Marketing-Head will ensure that proposal and estimation is prepared on time for all accepted RFPs.
  8. Effort and cost estimates are prepared during the proposal preparation stage. For any new project, CMG/DH will study the request, and estimate the effort, cost and schedule. For existing project, the DH/PM will estimate the effort, cost and schedule, and forward the same to the CMG and Account In-Charge.
  9. After project kick-off and identifying a Project Manager for the project, re-estimate the effort, cost and schedule for the project using Estimation Worksheet (TPPPL02/ TPPPL02a) After completion of SRS, the PM will re-estimate the effort, cost and schedule for the project using Estimation Worksheet (TPPPL02/ TPPPL02a).
  10. Re-estimation of effort, cost and schedule is carried out whenever there is any change request for the requirement, either from the customer’s side or by the project team itself. The SPMP is updated, reviewed and approved.
  11. The basis or justification of the schedule is recorded.
  12. The estimates will be prepared as per the Guidelines and using Estimation Worksheet (TPPPL02/ TPPPL02a). Please refer estimation guidelines to estimate effort, cost and critical computer resources (CCR).

17.3.2.8 Establish the Budget and Schedule (PP/SP2.1)

Establish and maintain the project’s budget and schedule.

Process Steps:

  1. List major and intermediate milestones in the project plan.
  2. Define the schedule. The dependencies between tasks should be captured in the project schedule. The assumptions and constraints associated with the schedule are listed.
  3. Define the budget using the template XX. The template will include a management reserve. The incremental funding requirements, if any, will be listed with the due date in the project plan.
  4. The project schedule and budget is approved as part of the project plan approval.
  5. The PM will present the schedule, effort, risks and the requirements from other support groups in the kick-off meeting.
  6. After completion of SRS the PM will re-estimate the effort and update the schedule for the project.

17.3.2.9 Plan for Data Management (PP/SP2.3)

Plan for the management of project data.

Process Steps:

  1. Determine the project data to be identified, collected, and distributed in the configuration management section of the project plan. Document the access privileges to project data in the configuration management section of the project plan.
  2. The PM, with the help of CTL will prepare and maintain the MLD FRPDM01 in PCC. The MLD will list both the "Internal" and "External" documents such as customer supplied documents and user manuals of the tools used for project execution.
  3. The documents will be reviewed and approved as per the Verification procedure (PR07). The approval authorities for documents will be as described in the SPMP for the project.
  4. The PM will ensure that the approved SRS, SDD, ATP, ATS, STP, STS, ITP, and ITS are put in the controlled library (baseline) of the project as per Configuration Management (PR-03) process area.
  5. The various records applicable to any particular project will be mentioned in FRPDM02. The PM will control access to quality records during project execution. If contractually required, access to quality records will be provided to the customer for the agreed period.
  6. Refer to Project Document Control for the list of configuration items and non-configuration items. The PM will create a Project Folder for the project as per Project document control (PR-01) . Use Project Kick-off Checklist CH35.
  7. Customer supplied material is handled as per the Control of Customer Supplied Material (PR-14).
  8. Document the mechanism to archive data, access to archived data and restoration of data in the backup and restore section of the project plan.
  9. The PM will maintain the master copy of project documents. Project documents will be archived as per the Archival Maintenance procedure.
  10. The PM will submit the quality records for archival as per the Archival Maintenance procedure. The suggested retention period for the quality records is three years from the date of archival or till the end of warranty, whichever is earlier, unless otherwise stated by the customer.

  5. Check for objective entry and exit criteria to authorize the initiation and completion of tasks.
  6. Check compatibility of the project plan with the stakeholder plan
  7. Include conflict resolution plan to manage stakeholders

Process Steps:

  1. Review the PPQA plan, Configuration Management plan and Risk Management plan.

Manage the involvement of the relevant stakeholders in the project.Participate with relevant stakeholders to identify, negotiate and track critical dependencies. Establish and maintain collaboration among interfacing teams

  2. Monitor and control project activities using the project plan, other plans and the project’s defined process. Check if entry and exit criteria are being satisfied for key tasks and phases.

  4. Monitor product and project interfaces
  5. Manage external and internal commitments

Every fortnight or monthly, depending on the duration of the project, align the project performance with current or planned needs, objectives, requirements of the Customer, organization and end users. Aligning performance can be achieved by adjusting project planning parameters and project risks.
  7. Changes to key configuration items, for example, Project Plan, must go through review and approval.
  8. Document the observations and results of managing the project in appropriate forms available in the QMS.
  9. Monitor project progress against plan every week or a defined periodicity based on the specific project. Internal stakeholders involved in the progress review meeting include the project team members, integrated team, quality assurance team and configuration management team. Document the issues, impacts and results of the internal stakeholder involvement progress review.
  10. Review of the Integrated team structure
  11. Review the team composition, any changes in team lead or team member, changes in team responsibility as part of the review of the integrated team structure.
  12. Specify the periodicity of review of the Integrated team structure
  13. Review the team composition, any changes in team lead or team member, changes in team responsibility as part of the review of the integrated team structure.
  14. Review the capability of the integrated team to meet project objectives based on initial staffing.
  15. Measure team performance, individual performance and critical success factors.

17.3.2.14 Monitor Project Risks (PMC/SP 1.3), Monitor Stakeholder Involvement (PMC/SP1.5), Monitor Commitments (PMC/SP1.2) and Monitor Data Management (PMC/SP1.4)

Monitor risks against those identified in the plan. Monitor stakeholder involvement against the project plan. Monitor commitments against those identified in the project plan. Monitor the management of project data against the project plan.

Process Steps:

  1. Revise the risks document or template when new risk is identified, when risks become problems, when risks are retired and when project circumstances change significantly.
  2. Project Manager will follow the risk management process/PA.
  3. Project team members will report potential risks
  4. Review the risks during weekly team meetings.
  5. Review risk management status with stakeholders during formal progress review meeting held monthly or on the planned event date or planned milestone date.

Monitor Stakeholders:

  1. Review the involvement of the external stakeholder every week or at a defined frequency or as per participation required in specific project activities. Document the issues, impacts and results of the internal stakeholder involvement progress review.
  2. Review the work products with the stakeholder to confirm if it is meeting the requirements of the stakeholder. Document the recommendations and actions to resolve problems in the product or deliverable requirements and design.

Monitor Commitments:

  1. List the commitments from external and internal stakeholders.
  2. List each of these milestones as part of the project schedule so that stakeholders can plan and attend. The stakeholders required to participate in each of these milestone meetings is listed in the project plan. MOM is circulated to all stakeholders after the meeting to document the commitments, actionable and action date.
  3. Review external and internal commitments every week in the internal and external stakeholder meeting. Review commitments that have not been satisfied and commitment having high risk of not being satisfied.
  4. Document the results of the commitment review.

Monitor Data Management:

  1. Changes to approved documents will be initiated based on the Change Request Form .The changes to the document will be reviewed as per the verification procedure. The reviewed document will be approved before execution.
  2. Ensure review data management activities are defined in the project schedule.
  3. The project team will enter data for measures associated with project issues.
  4. Identify and document significant issues and the impact.
  5. The SQA or Quality Department will review measurement data for accuracy and consistency.
  6. Document the results of data management activity review.

17.3.2.15 Conduct Progress Reviews (PMC/SP1.6), Conduct Milestone Reviews (PMC/SP1.7) and Monitor Project Planning Parameters (PMC/SP1.1)

Periodically review the project progress, performance and issues. Review the accomplishments and results of the project at selected project milestones. Monitor the actual values of the project planning parameters against the project plan.

Process Steps:

  1. Every week, the Project Manager will monitor tasks starting and ending dates, deliverables with content and quality, effort as planned, size of software work products as planned, costs as budgeted, critical computer resources as planned, risks, issues and actions along with the integrated team. Document the results of the review meeting in the minutes of meeting form.
  2. Every week, there is internal review project work products and activities, identifying any exceptions to the project plan and/or organization processes, review measurement data for accuracy and consistency, discuss deviations with project team and project manager and raise non-compliance items with management, if unresolved. Document the results of the review meeting in the minutes of meeting form.
  3. For change requests, document the change requested, along with priority of the change, (optional) approaches to handle the change and workaround if the change is not implemented. Enter change request into the change tracking system or/and change request template. Review the change request. Estimate the impact of the change on the project effort, cost, schedule, resources. Using the estimate, decide whether or not to authorize the change. If the change is denied, communicate that decision to the requestor and terminate this process. Incorporate the change into the project work plan and adjust resources and schedule as needed, to accommodate the change. Perform the work needed to address the change, and conduct the associated verification activities to ensure correctness. Update change request records to document the changes made and the completion status. Communicate completion status to the Change Control Board. Inform the requestor of the final status.
  4. Determine what kinds of information need to be exchanged. Decide what medium of communication is best – meeting, electronic exchange, other. Determine frequency of communication. Identify the stakeholders who will need to receive the information. Communicate status information, current action items, issues, risks, and status of technical activities.
  5. The Project Manager and the Configuration Management team member will accept items for configuration management as planned after checking if they meet the conditions set by the project and by organization processes for configuration management. Report on status and content of baselines.
  6. On reaching milestone, event, or date for review, the Project Manager identifies information that needs to be prepared and/or presented. Identify participants for the review. Establish tasks and assignments for the review. Establish review logistics. Prepare information for the review, including items such as list of accomplishments in last period, list of plans for next period, milestone progress reports (planned to actual), staffing profile (planned to actual), cost profile (planned to actual), size and critical computer resources (if appropriate), risk management status, issues, impacts, and action items status, quality assurance status , configuration management status, requirements management status, updated cost-benefit analysis, and milestones being met as planned. Document the results, action-items and decisions of the review meeting.
  7. The Project Manager determines that a significant change in the plan is needed. Changes can include life cycle approach, effort variance, schedule variance, change in requirements, or an alternate approach to the work, and tools and methods to be used are different from what was initially planned. Develop updates to project plan and review with all affected parties. The DH approves the updated plan. Establish commitments to the approved plan. Place the updated plan under configuration management.
  8. Track change requests, problem reports and action items to closure.

  7. Monitor and track corrective actions to completion
  8. Analyze corrective actions for effectiveness
  9. Update the issue log to capture the completion of corrective action and effectiveness of corrective actions.
  10. Track issues to closure.

Monitor and control the project planning, project monitoring and control and integrated project management process against the plan for performing the process and taking appropriate corrective action.

Collect work products, measures, measurement results, and improvement information derived from planning and performing the project planning process to support the future use and improvement of the organization process and process assets.

Monitor and control the organization process definition against the plan to perform the process and take appropriate corrective action.

The Project Monitoring and Control PA implements the generic practice of “Monitor and Control the Process”.

Refer to the Measurement and Analysis PA and record details related to measurement, recording, analysis and monitoring actual performance of the project planning, monitoring and control and integrated project management processes.

The specific practice of “Contribute Work Products, Measures and Documented Experiences to the Organizational Process Assets” can implement the generic practice “Collect Improvement Information”.

Process Steps:

  1. The Project Manager will list the project planning, project monitoring and control, and integrated project management activities in the schedule.
  2. The Project Manager will track the schedule, effort and cost variance and report the same in the weekly/fortnightly/monthly and other planned meetings.
  3. Record the schedule, cost and effort variance for every revision of the project plan.
  4. The Project Manager will identify project deviations based on established criteria that are documented in the Project Management guidelines. The number of tailoring options exercised is tracked and discussed with the quality department to ensure relevance and project success is not compromised.
  5. The approval of the revised plan is an indication of the approval of the revised schedule, cost and effort variance.
  6. The Project Manager will monitor and control the progress of the tasks assigned to create the work products and deliverables. Performance review will be conducted for both the project team and the integrated team.
  7. The Project Manager will monitor and control the project’s shared vision and effectiveness, integrated team structure and effective and integrated team structure and effectiveness. Expensive meetings, in case of international calls, video-conferencing, etc. are tracked and reported to ensure cost variances of the integrated team is under control.
  8. The Project Manager will monitor the timely execution of reviews, implementation of review comments and closure of review comments.
  9. Ensure Configuration Audit and Quality Audit is conducted so that work products, measure, measurement results and improvement information are compliant to organization processes.
  10. The Project Manager will ensure that the Configuration Items identified in the project plan are compliant to Configuration Management processes. The Configuration Items of integrated project management processes such as shared vision, integrated team structure and integrated team charter as defined in the project plan.
  11. The Project Manager will monitor and control the project using the measures and measurement results identified in the project plan and auxiliary plan such as the Quality Management plan.
  12. The Project Manager will monitor risks, probability of occurrence, impact and execute risk response measures.
  13. The project Manager will track the open and closed corrective actions. The results of corrective action will be documented. Interface coordination issues are tracked and closed.
  14. The Project Manager will track changes to the project defined process. The schedule and effort required to tailor the organization set of standard process is recorded. The project tailoring activities is planned and included in the project schedule.
  15. The Project Manager will track process and product measures and plan actions to ensure project success.
  16. The Project Manager will ensure that the improvement information, lessons learned, measurement data, change request to modify the organization standard process and non-standard tailoring undertaken in the project is added to the organization process assets.
  17. The Project Manager will ensure that the project complies with the processes obtained from the organization set of standard processes.

17.3.2.19 Verification (Objectively Evaluate Adherence and Review Status with Higher Level Management GP2.9 and GP2.10))

Objectively evaluate adherence to the project planning, project monitoring, project control and integrated project management processes against its process description, standards, and procedures and address non-compliance.

Review the activities, status and results of the project planning process, project monitoring and control processes and integrated project management process with higher level management and resolve issues.

Project will collect work products, measures, measurement results, derived from project planning, project monitoring and control, and integrated project management activities to support the future use and improvement of the organization processes and process assets.

Refer to the Process and Product Quality Assurance (PPQA) PA to implement and support the generic practice “Objectively Evaluate Adherence”. Record the details of the activities to objectively evaluate adherence in the project plan.

Refer to the specific practices of “Conduct Progress Review” and “Conduct Milestone Review” in the Project Monitoring and Control PA. Involve senior management in the progress and milestone reviews. This will support the generic practice of “Review Status with Higher Level Management”.

Process Steps:

  1. The Project Manager will review the implementation of the project defined process. The artefacts generated using the project defined process will be listed in the Configuration Items section of the project plan and serve as evidence of process compliance.
  2. The Project Manager will review the project planning activities, project monitoring and controlling activities and integrated project management activities such as establishing estimates, developing the project plan and obtaining commitments to the plans.
  3. The Project Manager will ensure the review and approval of project planning, project monitoring and control and integrated project management artefacts such as Estimation Sheet, WBS, Quality Assurance Plan, Configuration Management Plan, Risk Management Plan and Project Plan.
  4. The Project Manager will be guided by the shared project vision and use Integrated teams. The integrated team structure, team charter and shared vision statements will be recorded to serve as evidence of process compliance.
  5. The Project Manager will obtain commitment to the project plan from all project Stakeholders and record it in the minutes of meeting.
  6. The Project Manager will identify and collaborate with all relevant stakeholders for successful execution of project tasks. The minutes of meeting, actions planned, action taken, results, and effectiveness will be tracked using appropriate forms, communicated to stakeholders, reported to Sr. Management in the planned meetings and archived to serve as evidence.
  7. The Project Manager will record significant deviations from the plan using appropriate criteria and take corrective action. The results of the corrective action are documented.
