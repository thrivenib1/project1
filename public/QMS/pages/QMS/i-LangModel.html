Introduction

Any comprehensive software is expected to be user-friendly across the globe. Localizing the software into multiple languages and creating technical documentation for the software are inherent parts of software development. Integra has been involved very actively with various IT companies across the globe, in Localization and technical documentation for various SW applications. Hence, a life cycle model, called i-LANG model, (Integra Language Model) is developed to carry out such activities.

The basic framework of this model is shown in the figure above. This model will be applicable for both localization and documentation activities.

The development framework has four stages: Input, Transform, Verify and Output. Every work item goes through these four stages. The input is processed, and the “transformed” item is verified to examine whether it meets the implicit and explicit requirements. If it meets the requirement, it is passed to the Output stage; else, it is looped back to the Transform stage. This process is repeated until the work item meets the quality requirements. From the Output stage, a feedback is sent to the Input stage to improve the process.

The lifecycle is divided into five phases;

1. Requirement,

2. Analysis and Design,

3. Development,

4. Verification and

5. Delivery.

These fives phases are shown in Fig – 2. In each phase, the basic development framework (Figure –1) is applied. This model provides the same basic framework for technical writing as well as localization activities. The way the model is applied in each case, for each phase is explained in the following sections.

1 Requirement Phase

1.1 Requirements

On signing the contract the CMG will create PCC entry for the project. The Marketing In-Charge of the project or the PM of the project will call for project kick-off meeting. (Refer to ,FP18 Business Development Management.) The PM will present the schedule, effort, risks and the requirements from other support groups in the kick-off meeting. On formation of the project team, work on requirement phase begins. The PM will create a Project Folder for the project as per FRPDM01. Refer to MP01 Document Control Procedure. Use Project Kick-off Checklist, CHPPL04.

The first step in any developmental effort is to determine exactly the scope of the project. The requirements stipulate the inputs and outputs. Any standards or specifications to be followed are also stipulated in this phase.

The requirements are documented in explicit terms and approval is obtained from the customer, through e-mail or hand-written confirmation. The customer-accepted requirement is baseline. Customer approval record is put in the project folder.

1.2 Software Project Management Plan

The PM prepares the SPMP by using the PMP (TPPPL03). The PM can tailor the SPMP to meet specific project requirements with approval from the SEPG. Deviation, if any, will be mentioned in the SPMP.

The DH reviews and approves the SPMP to ensure that the commitments stated in the plan are pragmatic and can be met. The review is done through PCC / IRR (FRVER01).

The SPMP has to be updated under the following conditions:

1. Re-estimation of the project

2. Major scope changes to the contract

3. Formal / Informal reviews and audits (Internal/External)

4. Change request for requirements

The Revision History section in the SPMP will reflect the revisions made in the SPMP, and all the revisions require approval from the same authorities that reviewed and approved the earlier version of the plan.

If the multiple cycles of work needs to be carried out under the same project the PM will update the schedule and effort section of the SPMP and get approval of the same.

1.3 Resource Procurement

· The PM requests for human resources using PCC as per the PR11 Infrastructure and Human Resource Request and Release procedure

· The DH/CMG reviews the requests, and allots the human and infrastructure resources as per the requirement.

1.4 Project Induction

➣ The PM organises project induction for all the team members. Induction covers: Project overview, Process Model, client overview, milestones, and responsibilities.

➣ The PM invites the DH, Quality Manager and other members as per the need.

➣ The meeting is to be organised through PCC. If not possible, records of attendance (FRMMR01) and MOM (FRMMR02) will be kept in the project folder.

1.5 Training

Technical training as identified in the SPMP is conducted. The PM makes a training request to the Training Co-ordinator using FROTR01 for project specific training requirements.

1.6 Baseline

At the end of this phase, the PM ensures that the following configuration items (CI) are baseline using the Configuration Management procedure (PR04):

· Requirements document

· SPMP

· Customer supplied material which is used as input

1.7 Customer Supplied Material

All customer-supplied material is handled as per the Control of Customer Supplied Material (PR12). The customer-supplied material can be: Translation guidelines, sources in the form of documents and resources, and glossary lists

All the customer-supplied material is to be listed in the Master List of Documents through PCC/ FRPDM01. Refer to Document Control procedure MP01 .

2 Analyses and Design Phase

2.1 Analysis

The requirement document contains all the requirements of the project. It communicates the customer’s needs, wishes, and expectations.

The requirement document forms the basis for determining whether the developed system is complete. The verification of the system against the requirement document serves to demonstrate that the contractual agreement between the customer and Integra has been met, and generally implies the end of the development phase. This verification requires the construction of test situations i.e. ATP to objectively prove the system compliance with the requirements specified.

All requirements identified during the Analysis phase are documented in the Requirements Specification (RS). The main elements of an RS are: List of deliverables, list of languages to be localized into, and translation guidelines and glossary lists to be used (if any)

The Requirement Specification document is expected to be brief.

2.2 Re-estimation

After completion of RS the PM re-estimates the effort and the schedule for the project. The estimation is revalidated after preparing a detailed WBS for the entire project and recorded using Estimation Worksheet TPPPL02 . (Also refer to Business Development FP01)

Re-estimation is also carried out whenever there is any change request, either from the customer’s side or by the project team itself. The SPMP is modified to include all the affects arising out of re-estimation. The same person that approved the earlier version of the SPMP approves it once again, and the revision history is updated in the document.

2.3 Test Plans

In case of the existence of a Master Service Agreement, in which the acceptance conditions are mentioned clearly, and the specific project has no deviation from the standard project, creation and approval of the ATP from the customer can be omitted.

Checklists are prepared for checking the correctness and completeness of the project work item. Generic checklists are reviewed and improved to cater to the needs of the specific project. Special checklists also can be created to verify adherence to standard localization glossaries. This will be mentioned in the SPMP.

2.4 Baseline

The PM ensures that the approved RS, and the checklists are put in the controlled library (baseline) of the project as per Configuration Management procedure PR04.

3 Development

Documentation

3.1 Creation of the document

TM will create the document based on the document design. Requirements such as style sheets, writing style guidelines and file formats will be adhered to. In the absence of standards provided by the customer, Integra’s standards are followed. – Documentation guidelines (GLDES03, GLDES04, GLDES05). The checklist will be used for verifying the work item.

The Architect will provide guidance on technology and domain related issues during the preparation of documents.

· Self-review: After completing the work item the team members will do a self-review and fix the defects. The defects need not be logged, but all defects must be removed.

· Peer-review: One or more team members at a peer level will review the document for content, language, formatting and conformance with customer standards as well as Integra’s standards. The IRR module of PCC/ FRVER01 is to be used for recording the defects.

3.2 Translation

The translators will translate the source documents or resource files and will follow the guidelines provided by the Customer or specified by Integra, as per RS.

Translators will do a self-review of their work items to check for:

➣ Completion of translation,

➣ Adherence to guidelines,

➣ Basic spelling and grammar checks,

➣ Adherence to normal translation conventions.

➣ After completion of self-review the work item will be put for peer review. Review record of peer review will be maintained using FRVER01.

If translation is done through subcontractors, the activities will be carried out as per Purchase procedure FP08

4 Verification Phase

Documentation

The Architect or PL reviews the documents created by the team members. Correctness, completeness, and clarity of the content, correctness of the language, presentation of the material and formatting of the document are verified using Technical writing Checklist CHLAN02.

In addition, adherence to the document design, the standards and guidelines, and technical writing conventions, suitability of the content and presentation to the type of document being created are also verified.

Localization

Team members of the localization team will verify the translations received from the translators for completion of the translation, adherence to the translation guidelines, and formatting of the documents. The verification is carried out as per the checklists particularly designed for the verification of the specific work. In absence of these, the team members will use Localization Checklist CHLAN01.

4.1 Verification

The verification activities involve:

➣ Verification of completeness of the translation

➣ Verification of correct replacements of segments/strings in the respective file.

Generally, integration will be required when:

1. Technical documents are created for multiple modules of a software by multiple technical writers

2. Document creation will be done in parts

3. Localization will be done in parts

4.2 Verification process

➣ The defects found will be entered in BCC.

➣ The PL will allocate the defects, logged in BCC, to team members for fixing the defects.

➣ TM will log into BCC to find the defects allotted to them, fix the defects and log the fix.

➣ The PL will review the changed work items, and will follow up to closure.

➣ After completion of verification process the work item will baseline by the CTL/ PM.

➣ A release readiness check will be done, and when all the defects are closed, the work items are sent for delivery.

5 Deliveries

5.1 Packaging and Delivery

After successful completion of the testing phase, the PL will generate the DN (Delivery Note) and RN (Release Note) by using PCC / Delivery Note (FRPRI03) and sends them to the PM along with the System Test Report.

The SQA team will verify that all deliverables have passed through the required tests and the verification processes by carrying out a release readiness check. This has to be approved by DH

The Deliverables will be are packed as per the contract agreement and sent to the customer.

The PM will request the commercial dept for raising the invoice as per the agreement.

5.2 Acceptance Test

Acceptance test is carried out in the customer’s place/environment as per the acceptance test plan.

Defects found during the acceptance test are logged using BCC or FRVAL01 or in customer’s own format and fixed and installation plan will be updated accordingly.

5.3 Customer Signoff

Customer Signoff will be obtained after completing the Acceptance Test successfully. (Refer Business Development FP01 )

5.4 Warranty Support

Deliverables are supported as per the contract.

5.5 Project Closure

Infrastructure will be released as per Infrastructure and HR request and Release procedure PR11 .

Customer supplied material will be returned as per Control of customer supplied material PR12 .

The PM collates project metrics (Refer to FP11) and calls for Project Closure meeting as per PR15 Project Closure Procedure.
