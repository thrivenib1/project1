Configuration Management
ISO Clause: 7.5 PR04 Version: 3.00 Date: 17th May 2010

4.0 Introdution

4.1.1 Purpose
This process defines the method used within Integra for Configuration Management (CM) of work products. CM helps in establishing and maintaining the integrity of work products using configuration identification, configuration control, configuration status accounting, and configuration audits.

4.1.2 Policy
All work products developed as part of the product development or service delivery will follow the configuration management procedure.

4.1.3 Scope
The scope of Configuration Management involves identifying controlling, and tracking all versions of hardware, software, documentation, processes and procedures. This Process Area discusses the main processes and activities involved in configuration management for projects throughout their life-cycle.

4.1.4 Entry criteria

  • Project Kick-off and Configuration Management Plan.
  • Training for the stake holders on the level of knowledge required to work with the tool

4.3.2 Tasks
The steps of the CM process are described below:

4.3.2.1 Identify configuration item

  • Configuration items (CIs) and Non-Configuration items (non-CIs) for projects will be identified and documented in the Configuration Management Plan based on the Contract, SOW, URS and other relevant document available at the start of the project. All work products that would exist and may require modification during the life time of the product will be treated as CIs.
  • Project Manager and Configuration Controller will be responsible for identifying the CIs and non-CIs for the project.

4.3.2.2 Establish a Configuration Management System

  • Project Manager will identify and allocate a resource as Configuration Controller (CC). The Configuration Controller could be also a person who is common for more than one project.
  • The CC will be responsible for the configuration management activities along with their well defined responsibilities in CMP.
  • CMP will be reviewed and duly approved by Delivery Head.
  • The dynamic and controlled directory structure along with the access control as per the approved CMP will be established to manage CIs and NonCIs.
  • Relevant team members will be responsible for all check-in and check-outs of the CIâ€™s in dynamic directory structure of the configuration management system of the project
  • The version numbers of documents will be maintained as per the Version Numbering procedure (MP01 Section 1.3.1.2.1)
  • Identify and document non-compliance

4.3.2.3 Create or Release Baselines

  • Reviewed and approved CIs will be baseline by Configuration Controller/Project Manager and thus will serve as basis for any further development.
  • A baseline record will be prepared by Configuration Controller using baseline record form (FRCON02) every time a CI is baseline.
  • Project Manager will review the baseline record for its completeness and correctness.
  • DH will authorize every release of baselines.
  • All the deliveries to the customer will be made from the baseline directory.

4.3.2.4 Track Change Requests

  • Changes for baselines could be raised internally or externally by customer. These changes could be indicated through E-mail, Fax or Telecom minutes. The changes could be of new or modified requirements or due to the result of document review or testing. The change request categories include Hardware, Software, Documentation, COTS packages and Operating System used in the current project.
  • Depending on the result of impact analysis CRs acquire the status as accepted, rejected, on hold/deferred, duplicate or forward.
  • All the deliveries to the customer will be made from the baseline directory.

3.3.2.5 Control Configuration Items

  • Accepted CRs will be assigned for resolution and corresponding work product is identified for implementation.
  • Configuration Controller ensures that the correct work product is delivered for implementing change request. Configuration controller will manage and control these CIs, version control of the documents will be done as per PR 13, Project Document Management procedure.
  • PM would review and ensure that the change request implementation has not caused unintended effects on the baselines

4.3.2.6 Establish Configuration Management Records

  • Completed CR will be verified for correctness of its implementation by PM/CC and will be marked as closed. Change Control Record will be accordingly updated or the same will be generated using PCC.
  • PM will track the changes to various CIs and maintain their revision history using Configuration Status Record.
  • CC will prepare Configuration Management Plan(TPCON01) for the latest versions of all CIs

4.3.2.7 Perform Configuration Audit

  • The QA member will conduct the Configuration Audits on CC and in his absence on Project Manager at the time of each baseline to verify the compliance to configuration management activities defined in the CMP for the project. The audit can involve other stakeholders, if required.
  • All non-conformances will be reported in a Configuration Audit Report (FRCON05), and tracked to closure by the QA member.
