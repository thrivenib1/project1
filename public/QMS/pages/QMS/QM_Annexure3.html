4. Process Description

Defining PDP (Project’s Defined Process)

All the Projects executed at INTEGRA shall follow the processes defined as part of the INTEGRA QMS, throughout the life cycle of the Project

INTEGRA has authorized MR (Management Representative) to make the necessary changes to the QMS (OSP) and is the custodian of all the process assets of INTEGRA. The process changes are defined and recommended by the SEPG.

Based on Project requirements the processes, methodologies, standards, guidelines and checklists may be tailored using the tailoring guidelines.

All the Projects of INTEGRA should follow the below mentioned processes with necessary tailoring as appropriate. Also, these processes are supported by other process assets such as templates, forms, and guidelines. Project should use these process assets.

SDLC Process
Process Purpose
Requirements Elicitation and Management To lay out a procedure that is to be followed in Projects to produce, analyze and manage requirements and to identify the inconsistencies between those requirements and Project’s plans and work products.

The process gives guidelines that help to come out with various designs such as High Level Design (HLD), Low Level Design (LLD), Unit Test Plan (UTP) preparation, System Test Plan (STP) etc.
Development

This process provides guidelines that one can use for coding, reviewing the code and unit testing. The major activities include:

· Program specification and UTP understanding

· Code development

· Code review

· Preparation for Unit Testing activities, like test bed creation, test data preparation etc.

· Unit testing

· Unit test result review and defects closure
Testing

This process is to provide guidelines for performing Integration Testing & System Testing using individually tested program units. The major activities include:

· Test data preparation

· Testing and test results logging

· Test Results Review



Support Processes
Process Purpose
Project Planning

This process helps to establish detailed plans for performing the various tasks of software engineering in a Project as also for managing it. The major activities of this process include:

· Re-estimation

· Resource allocation

· Risk Management (Through Risk Management Methodology of INTEGRA and Risk planning and management)

· Preparation of the plans for the Projects.

· Data management planning

· Project Kick-Off
Project Monitoring and Control

This process is aimed at giving the stakeholders adequate visibility to the progress and issues in the Project’s as also ensures the Projects are executed as per the plan. This process will give guidelines, for what metrics are to be collected so that managers of the project can take appropriate preventive and corrective actions.

Typically the Projects are tracked for:

· Size

· Effort

· Schedule

· Software and Hardware resources

· Quality of work products

· Risks

· Issues

· Commitments

· Training requirements

· The tracking of these parameters is done normally on a weekly basis, as well as on event driven basis.
Configuration Management

The purpose of this process is to establish and maintain the integrity of work products using configuration identification, configuration control, configuration status accounting, and configuration audits. This process consists of the following major activities:

1. Identification of configurable items

2. Preparation of Configuration Management Plan

3. Establishing baselines

4. Performing baseline audits

5. Change management

6. Status accounting
PCM Process

The purpose of this process is to ensure that the Quality system is established, implemented, maintained and continuously improved in accordance with INTEGRA best practices as well as in alignment with market proven models such as ISO and CMMi.
Review Process

The Purpose of this document is to describe the Review Process for all technical reviews conducted for the Projects executed in INTEGRA.

Some of the critical work products covered under the technical reviews includes:

· Project Plan/Operational Framework

· Project management procedure document(s)

· Configuration Management Plan

· Work specification document/Statement of work/ system specification/requirement specification document

· Design Document (High level design / Low level design)

· Test Plans / Test results

· Source Code

· User Manuals/ Delivery Documents

Three types of Technical Reviews are conducted at INTEGRA. These are Self-Review, Peer review, One-on-one review

Metrics Management

The purpose of Metrics Management Process is to describe all metrics related activities at INTEGRA. The activities include data collection, data analysis, corrective and preventive action to be taken based on the analysis done. This covers both process quality and product quality metrics.

All the metrics related activities are planned and executed at both Project level and at Organizational level to help the Project and INTEGRA to quantitatively control the process and product quality.

The major activities in this process are:

· Planning for the metrics suitable for the Project and the organization

· Data Collection, analysis & reporting

· Creation of process capability baseline
PPQA

The IQA (Internal Quality Audit) procedure is established to assure that software developed at INTEGRA complies with the processes defined in the QMS and provides visibility to the management about the procedures used & product built by the Project team.

Periodic Activities includes

Carry out Verification activities as per the IQA schedule.

· Report the findings of this verification, through IQA report

· Discuss and assist Project teams in resolving issues, taking corrective actions

· Analyze the defect trend & customer complaints and assist the team in carrying out Causal Analysis and taking preventive/corrective actions.

· Ensuring that Project Metrics are collected from Project’s.

· Conduct quality related awareness / training programs as per the requirements of the Project’s



Project -end activities includes

· Ensuring that Project end appraisals are completed for all the members

· Identify the best practices, areas of improvement and tools that were used

· Consolidate the metrics

· Initiate archival of Project documents/media
Closure and Archival process

The Purpose of this document is to describe the procedure to be followed when a Project comes to a closure. The end of Project is normally reached when all the Project deliverables are transferred to the client organization.

The main activities of this process include:

· Review and update the Project Work Plan with all required details.

· Review and complete contractual obligations, including:

o Intellectual property protection or transfer

o Licenses, royalties and copyrights

o Facilities transfer or dismantling

o Obligations to subcontractors

o Warranty periods;

· Confirm delivery completion to INTEGRA accounting services

· Conduct the Project end audit

o Problems / challenges faced and how they are resolved

o Lessons learnt

· Review the contract assessment with the customer's representative:

o Ascertain customer satisfaction with project outcome

o Apply Customer Satisfaction Assessment Program Update the Project description form

· Archival

o The important documents of the Project should be archived periodically and after the end of the Project

Typically the following can be archived for Project’s:

· Retrieval

· Disposal of Records
Training Process The purpose of training Process is to continuously upgrade the knowledge of the individuals to enable them to perform their roles effectively and efficiently.
Decision Analysis and Resolution Process

Decision Analysis and Resolution Process (DAR) is used at INTEGRA for Tool evaluation, Technical alternatives, senior management strategic decisions (such as Bid / no bid decisions, Vendor selection, Critical resource allocation / role allocation), Process improvement proposals evaluation and any other decision for Project’s where cost of reversal of decision is heavy or irreversible decision or client sensitivity.

The purpose of Decision Analysis and Resolution (DAR) process is to analyze possible decisions using a formal evaluation process that evaluates identified alternatives against established criteria.
