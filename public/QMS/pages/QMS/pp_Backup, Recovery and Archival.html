Backup, Recovery and Archival
ISO Clause: 7.5.5 PR10 Version: 3.00 Date: 17th May 2010

10.0 Introduction

10.1.1 Purpose

The purpose of this procedure is to define the mechanism for backup of all work products.

10.1.2 Policy

This process is applicable for disaster recovery of the data relating to projects. The system administrator will maintain backups of data as stated in this process.

10.1.3 Scope

This procedure is applicable to project software items on the following:

  • Desktop PCs
  • Common server

10.2 Stakeholders

10.2.1 Responsibilities

Sl. No. Role Responsibility Authority
1. All Team Members Prepare Desktop PC for Backup NIL
2. PM / PL Prepare Project Server for Backup
3. System Administrator Backup Common Server and Restoration Check Approve backup
4. General Admin In-Charge Maintain / Store / Archive & Issue Control of archive Approve access to archive

10.3. Task Details

10.3.1 Backup

10.3.1.1 Prepare desktop PC for backup

The project team will plan for their backup activities based on the CMP.

All team members will prepare the source items / other project documents for backup by copying them on to the designated area on the Common server by end of the day. The source items can be compressed, if required.

10.3.1.2 Prepare project server for backup

  • The project team plans for their backup activities based on the CMP.
  • The PM /PL will copy the controlled and dynamic libraries on the project server to the designated area on the specified common server everyday before end of dayâ€™s work. The source items can be compressed, if required. Project backup is taken by the PM/PL before end of the day by copying the controlled and dynamic libraries from the configuration management tool into RWCD / RWDVD or a specific location in their machine. The PM/PL will retain the backup for only one working day.

10.3.1.3 Backup common server

  • Weekly backup: On the last working day (Friday) of the week, the system administrator will back up the common backup server in to CDs/DVDs.
  • Monthly backup: On the last Friday of every month, the System Administrator will back up the common backup server in to CDs/DVDs. These CDs/DVDs will be stored off-site to ensure retrieval of data in case of any disaster happening at the development site.
  • The following label format will be used to identify the backup CDs/DVDs:
    • Backup Date: CD No.: Contents:
    Integra
  • Multiple CDs/DVDs can be used to perform a full backup if data cannot be accommodated in a single CDs/DVDs and CDs/DVDs will be labelled as x of y. (For example, CD/DVD 1 of 3.)
  • Once the backup is done, a record is created in the Backup Log.

10.3.2 Restoration

After taking the backup the system administrator will restore the backup files to verify whether the backup files are retrievable or not. A new backup will be taken if it is not possible to restore from the first backup.

10.3.3 Archival

10.3.3.1 Prepare package for archival

  • The PM will prepare the package for archival by copying the source items to the archival media.
  • The entire repository of the source should be prepared for archival.
  • The source items can be compressed before copying, if required. The Project Folder, Quality Folder and Configuration Folder will be archived through suitable archival media.
  • The PM will prepare the list of items going for archival.
  • The archived material of the project will be indexed using Archival Register (FRARC01).
  • The index No will be A-yyyy-nnn where, yyyy is the archival year and nnn is a running serial no. 'nnn' will start from 001 in the beginning of the year.
  • The Index No will be written on the media on which the archival is made.
  • All project documents in Hard copy format will be converted into electronic format before archival and the hard copy will be destroyed.

10.3.3.2 Maintain/Storage of Archival

  • The General Admin will make an entry in the Archival Register (FRARC01). when an archival is received and will maintain/store the archival in a location so as to prevent physical and environmental damage. For any reason the archival for the internal projects must not be destroyed.
  • The General Admin will, at the beginning of every year (January), identify the external project archives that are more than three years old from the date of archival and destroy them.

    • 10.3.3.3 Issue Control of Archival

    The General Admin will maintain the record of archival issue and return of archival using Archival Access Request (FRARC02). All the Archival Issue requests need to be closed within 5 working days after the issue request. If the Archival Issue requests is not closed within 5 working days then Admin team has to follow up the person who has taken the Archival and ask him/her to close the request or if it is needed for some more time then extend the Archival issue period after getting the justification. If the archival is needed for a period of more than five days it should be specified along with the justification.
