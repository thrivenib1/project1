6.0 Introdution

6.1.1 Purpose
This process defines the procedure used within Integra for providing Technical Solution (TS) for internal and external projects.
The purpose of Technical Solution is to design, develop, and implement solutions to requirements as specified in the SRS. Solutions, designs, and implementations encompass products, product components, and product-related lifecycle processes either singly or in combinations as appropriate.

6.1.2 Policy
All software development projects will develop design specification for the project as stipulated in this process document.

6.1.3 Scope
This process is applicable to all product, product component, product related life-cycle process across the organization.

6.1.4 Entry criteria
The basic criteria for initiating the TS process are:

  • RFP
  • SRS

6.3.2 Tasks
The steps of the TS process are described below:

6.3.2.1 Develop alternative solutions and selection criteria (SP 1.1)

  • The architect will prepare the DOU using TPREQ01
  • The screening criteria of the alternate solution will be defined and documented in the DOU.
  • The architect will develop alternate solution for the requirement document in the DOU.
  • Identify and evaluate if any COTS products can be used in the solution architecture.
  • The architect will define the selection criteria of the alternate solution and document in the DOU.
  • A requirement traceability matrix will be prepared using TPREQ02 to ensure the traceability between the requirements and the alternate solution.
  • The DOU will be reviewed and approved internally by the BUH or peer architect.
  • Verification procedure, PR07 will be used for the review process.

6.3.2.2 Select product component solution (SP 1.2)

  • Based on the evaluation of alternative solutions the architect will update the selection criteria as required.
  • The architect will identify the re-usable components to be used/developed in the project [B1].
  • The architect will prepare the document of understanding (DOU) TPREQ01 for the proposed solution and obtain an endorsement from the customer.
  • The DOU is sent to the customer after review by a peer architect.
  • A requirement traceability matrix will be prepared by the architect using TPREQ02, which will have the traceability of requirements to the proposed solution.
  • All the work products viz DOU, RTM and re-usable component document[B2] will be reviewed and approved before base lining them in the project repository[B3].

6.3.2.3 Develop design for the product or product component (SP 2.1, 2.2 and 2.3)

6.3.2.3.2 HLD

  • The architect will prepare the HLD keeping the following in mind:
    1. Use data flow diagrams and control flow diagrams to diagrammatically represent the design.
    2. Make the design modular by adapting suitable design methodology like object oriented design.
    3. Update the traceability matrix to establish traceability between software requirements and HLD.
  • The input for preparing the HLD TPDES01 would be SRS, DOU, re-usable comp. doc, design guidelines.
  • The architect will call for a review seminar to review the HLD and baseline the HLD after approval. Refer to verification procedure for more detail on review process.
  • The HLD will be baseline after approval.

6.3.2.3.3 LLD

  • The technical lead of the project will be responsible for preparing the LLD TPDES02 with the help of the team members.
  • Document the low level design using diagrammatic representation like sequence diagram and class diagram.
  • Identify the unit test cases required to test the code.
  • The technical lead will update the RTM.
  • Identify coding language and appropriate coding standards to be used in the project.
  • The LLD will be reviewed and approved by the architect.
  • The LLD will be baseline after approval.

6.3.2.3.3 Perform make, buy or reuse analyses (SP 2.4)

  • The architect will perform make buy analysis.
  • The make buy analysis will be reviewed by peer architect before taking the final decision

6.3.2.5 Implement the design (coding) (SP 3.1)

  • The project team members will code as per the LLD in the identified language/s.
  • All the algorithms, data structures, data schema are converted into code.
  • Coding standards will be followed while developing the code.
  • Peer review of the code will be carried out and code review records will be maintained.
  • The team members will test the units as per the unit test cases identified. The team members will establish the traceability of code with LLD at class/file level with appropriate file headers and function headers as necessary.

Refer Verification procedure for further details on reviews and testing.

Refer product integration process (PI) for integrating components together to form the system.

6.3.2.6 Develop Product Support Documentation (SP 3.2)

  • Develop necessary support documents for the end product that helps rebuilding, installation, and usage of the system with the help of the technical writers.
  • The technical documents may include the following as applicable:
    1. Build notes
    2. Installation guide (TPDES03)
    3. User manual (TPDES04)
    4. Online help (TPDES05)

The PM will assign the tasks to the technical writers.
