Introduction

A software development process is a set of actions, tasks and procedures involved in producing a software system, throughout its life cycle. The objectives of software development management are to produce according to plan and to improve the capability to produce quality software solutions within the given schedule and budgeted cost. Keeping this in mind Integra has developed a software development lifecycle model called

i-FI Model (Integra Fractal Integrated Model).

The basic framework of this model is shown in the fig above where, a software development activity is divided into four different stages: Input, Transform, Verify and Output. These four stages are applied to every work item in the software development lifecycle and at the end of a particular work item, the “transformed” item is verified to examine whether it meets the stated and implied quality requirements. If it meets the requirement then it is passed on to the Output stage. If the work item fails to meet the requirement then it is looped back to the Transform stage and the process is repeated until the work item meets the requirements. From the output stage a feedback is created to the Input stage as a means of providing improvement mechanism to the entire process.

In this model the software development lifecycle is divided into six phases:

1. Requirement,

2. Analysis and Design,

3. Development,

4. Testing

5. Delivery and

6. Warranty

These six phases are shown in fig-2. The basic development framework (fig-1) is applied to different phases of the development lifecycle leading to a fractal. This ensures quality of the delivered software.

After the software is delivered, feedback is collected from the customer to identify improvement opportunity in the process. This process model will be followed in all development projects of duration more than eight weeks. For small projects, with duration less than eight weeks, the PM can request the EPG for process tailoring. In small projects, at least PMP, CMP, QMP, RMP, SRS, STP, Test Specifications documents will be prepared, reviewed and the review record maintained. The PM will ensure that the code is reviewed.

Brief discussion and detail activities involved in each of these phases are given below:

Requirement Phase:

On signing the contract project entry will be made in the PCC. The sales partner for the project will call for the project initiation meeting of the project and prepare the PIN using FRPPL01. The PM of the project will call for project kick-off meeting within one week of project initiation. The kick-off meeting can be combined with the project initiation meeting. Use downloads\QMS\ifi\CHPPL04.docProject Kick-off Checklist, CHPPL04. the PM will present the schedule, effort, risks and the support required from other support groups in the kick-off meeting using the Project Kick-off presentation template TPPPL06. On formation of the project team, work on requirement phase begins. The PM will create a Project Folder for the project as per Control of Document (MP01).

1.1. Project Management Plan

1.1.1. Preparation of PMP

The PM will prepare the Project Management Plan (PMP) by using the PMP template (TPPPL03), CMP using the template TPCON01, QAP using the template (TPPPL04) and RMP using the template (TPRSM01). The PM can tailor the PMP to meet the project requirement with approval from the EPG. (For multiple projects, the PM will prepare a sub-project plan and customise the documents as per requirement for the project.

1.1.2. Review & Approval of PMP

The BUH will review and approve the PMP, CMP, QAP and RMP and sub-project plan (if applicable) to ensure that the commitments stated in the plan are pragmatic and can be met. BUH will record the review findings using the Inspection and Review Record (FRVER01) and the record will be kept in the Project folder /PCC. Please use checklist CHPPL02 CHPPL-03 CHRSM01.doc while reviewing the PMP, if the review is not done using PCC.

1.1.3. Amendment to PMP

The PMP, CMP, QAP and RMP have to be updated under the following conditions:

➣ Re-estimation of the project occur

➣ Major scope changes to the contract occur

➣ Changes occur due to Formal / Informal Reviews and Audits (Internal/External)

➣ Changes occur due to Change Request

➣ Changing the risks in the project

The revisions made in any of the above documents will be reflected in the Revision History of the respective documents and will require approval from the same authorities that reviewed and approved the plan initially.

1.2. Resource Procurement

The PM will request for infrastructure and human resources as per Infrastructure and Human Resources Request and Release (PR11).

1.3. Project Induction

The PM will organise Project Induction for all the team members. Induction will be conducted in two parts:

➣ Part A will cover project overview, process overview, client overview, project milestones and responsibilities. The PM will invite Documentation team member and PQA.

➣ Part B will cover technical training as identified in the PMP. The PM will raise a training request using (FROTR01) to the Training Coordinator.

Attendance Sheet (FRMMR01) and MOM (FRMMR02) of the project induction will be kept in the project folder.

1.4. Software Requirements Specification (SRS)

The first step in any software developmental effort is to determine exactly what the software system will do. Software Requirements stipulates what must be accomplished, transformed, produced or provided. Software Requirement is a software characteristic that must be met or possessed by a software component in order to satisfy a contract, standard, or specification.

The requirements development procedure, PR05, is used to develop the requirements and document them in the SRS using the template, TPRQD02. Use guidelines GLRQD01, while preparing the SRS. It communicates the customer’s needs, wishes, and expectations to the developers of the system and may include descriptions of:

➣ The required computational functionality

➣ A description of the system behaviour

➣ Guidelines for the user interface

➣ Technical aspects of the Hardware / Software Interface

➣ Operational characteristics

Software Requirements also form the basis for determining when the software is complete. The verification of the software functionality against the requirement documents serves to demonstrate that the contractual agreement between the customer and developers has been met and generally implies the end of the development phase. This validation or validation and verification requires the construction of test situations i.e. ATP to objectively prove the software product compliance with the requirement document. The ATP and ATS are prepared and given by the customer. In the event that the customer does not give the ATP and ATS then, the STP and STS would form the basis for acceptance of the software.

The ownership of development of the SRS lies with the Architect. However, the Architect can take help of any team members of the project.

After completion of the SRS document, the PM will call for a seminar and present the SRS to the expert group of people for review. The Review leader (BUH) will identify a recorder and get the review finding recorded as per Verification Procedure (PR07) and the record will be kept in the Project Quality folder. The PM will ensure the closure of the action items of review finding. PQA will verify that all action items are resolved / closed before final approval of the document. SRS checklist CHRQD02 will be used while reviewing the SRS. If PCC is used then the PCC checklist will be used for the same.

1.5. Test Plans

The testing team members will prepare the STP (Software Test Plan) and STS (System Test Specification) for the project. The Architect will review the STP and STS and review record will be maintained. Refer to

the Validation Procedure (PR08). Use guidelines for STP (GLVAR01 Test Planning) and STS (GLVAR02 Test Requirement) while preparing the documents.

1.6. Re-estimation

After completion of SRS the PM and Architect will re-estimate the effort and the schedule for the project. Responsibility of re-estimation lies with the PM. The estimation is revalidated after preparing detailed WBS for the entire project and recorded using Estimation Worksheet (TPPPL02).

Re-estimation is also carried out whenever there is any change request for the requirement, either from the customer’s side or by the project team itself. The PMP is updated to include all the affects arising out of re-estimation. The same person that approved the PMP earlier approves it once again and the revision history is updated in the document.

1.7. Baseline

At the end of this phase the PM will ensure that the ATP, ATS, SRS, PMP, CMP, QAP, RMP, Estimation Worksheet, STP and STS are baseline as per the Configuration Management procedure (PR04) .

1.8. Customer Signoff

The PM will get the SRS and STP approval from the customer, through either e-mail or written confirmation.

1.9. Tailoring guideline

The process tailoring takes place in the Analysis and Design phase.

a. For projects with duration less than eight weeks, the HLD and LLD can be combined into one design document. The PM will ensure that PMP, CMP, QAP, RMP, SRS, design document is prepared and reviewed. The code is reviewed and review record maintained.

b. For small projects, with duration less than six weeks, the PM can request the EPG for process tailoring. In small projects at least PMP, CMP, QAP, RMP, SRS, STP, Test Specifications documents will be prepared, reviewed and the review record maintained. The PM will ensure that the code will be reviewed and review record will be maintained.

1.10. Customer Supplied Material

All customer-supplied material will be handled as per procedure for Customer Supplied Material (PR12).

2.0 Analyzes and Design Phase:

2.1 Analysis and design

Software design is an iterative process through which requirements are transformed into a blueprint for developing the software. It focuses on four areas:

➣ data structure,

➣ architecture,

➣ interfaces,

➣ components.

The Architect will ensure development of the High Level design document using TPDES01 and Low Level Design Document using TPDES02 . Refer Guidelines for Software Design Document (GLDES01). The technical solution procedure is used to develop and document the HLD and LLD.

The ownership of development of the HLD lies with the Architect. However, the Architect can take help of any team members of the project to develop the HLD.

The PM will then call for a review seminar of the HLD. The Review leader (BUH) will identify a recorder and get the review finding recorded as per Verification procedure (PR07) and the review record will be kept in the Project Quality folder. The PM will ensure the closure of all action items of review finding. PQA will verify that all action items are resolved / closed before final approval of the document. While reviewing the HLD, review checklist, CHDES01 will be used. If PCC is used for the review, then PCC checklist will be used for the same.

The responsibility of development of the LLD will lie on the PL of the project.The architect will review and approve the LLD before using it further in the SDLC.

The PM will consider process tailoring in the event that the project duration is less than eight weeks. Prior approval from the EPG will be sought in this regard. For all projects of duration less than eight weeks the HLD and LLD can be merged into one design document.

The HLD and LLD will be moved to the baseline folder in CVS after these documents are reviewed and approved.

2.2 Test Plans

During this phase the ITS is developed, reviewed and approval is taken from the concerned authority.

2.3 Customer signoff

The PM will get the customer approval on the HLD if required contractually. Customer approval will be kept in the Project folder.

2.4 Baseline

The PM will ensure that the approved HLD, LLD and ITS are moved to the baseline folder in CVS. Also the updated PMP, QAP, CMP, RMP and Estimation worksheet of the project will be baseline after updating the documents as applicable.

3. Development

Development Phase

3.1 Unit Test Plan

➣ Baseline LLD and STP are taken as input in the development phase

➣ The PL allocates work to TM (Team Member) to develop the UTP and UTS on the basis of the LLD and the STP.

➣ After peer review of the UTP, UTS PL reviews the document and the same is moved to the baseline folder in CVS.

➣ Review is done as per Verification Procedure (PR07) and review record is maintained (FRVER01)

3.2 Coding and code review

➣ Team member generates code using the LLD, as per the coding standards.

➣ Conduct self-unit test and fix defects. (Here the team member fixes the defect and the defects need not be logged, but all defects must be removed.)

➣ The peer group reviews the code for conformance against coding standards, program logic, algorithms and any other project specific standards.

➣ Code Review is done as per verification procedure (PR07)and review record is maintained (FRVER01)

3.3 Unit test

➣ The peers carry out unit Test. Test is conducted as per the Unit Test Plan (UTP) and Unit Test Spec. (UTS).

➣ Unit test is carried out as per verification procedure and the defects are logged using report (FRVER02)

➣ The programmer fixes the defect.

➣ The code is Unit tested once again as per unit test plan till all defects are removed.

➣ The UTSC will be moved to Test Library by CTL

4. Testing Phase

4.1 Integration Test:

➣ The PL gives ITS from the controlled library and unit tested source code from the Test library of the dynamic area to the Team Member.

➣ Team member performs integration test and records all defects found during integration testing in the Integration Test Report FRPRI02 /BCC

➣ Team member fixes bugs found during Integration Test.

➣ The PL will review the changed code using IRR (FRVER01).

➣ Integration test is conducted once again after fixing all the defects to ascertain that the software is defect free.

➣ The PL will review the Integration test report and will ensure that all defects are removed.

➣ After the completion of Integration test the CC will put the Integration tested source code into the Baseline Directory.

4.2 System Test:

➣ System test carried out as per the Validation Procedure PR08

➣ The TL gives Integrated Source Code, and the STP, STS along with the User Manual (Optional) (developed by the Documentation Group, reviewed and approved by the PM), to the Testing team from Controlled library for System Test.

➣ The bugs found are entered in BCC (Bug Control Centre) of PCC (Process Control Centre).

➣ The PM/ PL will do impact analysis using Software Problem Report FRCON07 (Ref PR04) /BCC and allocate bugs to team members for fixing.

➣ Team member logs onto the BCC to find out the bug allotted to them and they fix the bug.

➣ The PL will review the changed code and record defects using IRR (FRVER01).

➣ The PL will review the System test report and review record is kept using FRVER01/PCC

➣ The Architect reviews the design and implementation of the system based on the test results and defects identified.

➣ When the Software passes all the requirements then it is sent for delivery.

5. Delivery

5.1 Packaging and delivery

➣ After successful completion of the test phase, the PL will generate the DN (Delivery Note) using FRPRI03 and RN (Release Note) using PCC and sends them to the PM along with the Installation Pack, User Manual (Optional) and the System Test Report.

➣ The PQA will verify that all deliverables have passed through the required verification and validation processes.

➣ The release readiness review checklist will be used while conducting release readiness check of a project.

➣ All deliveries to the customer will be made from the baseline directory.

➣ Final inspection and Release readiness Review of the Package will be carried out.

➣ The Deliverables are packed as per the contract agreement and sent to the customer.

➣ The PM requests the commercial dept for raising the invoice as per the agreement.

5.2 Acceptance test

➣ Acceptance test is carried out in the customer’s place/environment as per the acceptance test plan.

➣ Defects found during the acceptance test are logged using BCC / Software Problem Report (FRCON07) , impact analysis is done and Defects are fixed.

➣ The PL will review the changed code using IRR FRVER01 .

➣ Regression test will be performed before sending the fixed code to the customer.

➣ Installation plan is updated accordingly.

➣ Any training necessary to operate the software is provided to the concerned groups.

5.3 Customer Complaint

Customer complaints will be handled as per Customer Complaint Management procedure PR14 .

5.4 Customer Signoff

Customer Signoff is obtained after completing the Acceptance Test successfully. (Refer to the PMP of the project).

5.5 Archiving

The Installation pack is then archived as per the scope of contract agreement as per Backup, Restore and Archival Maintenance (PR10) .

5.6 Project Closure

➣ Infrastructure and Project manpower is released as per Infrastructure and HR Request and Release - PR11

➣ Customer supplied material is returned as per the Control of Customer Supplier Material (PR12).

➣ The PM collates Project metrics (Ref FP11) and calls for Project Closure meeting as per Project Closure procedure (PR15).

6 Warranty Support

➣ The installed software application is supported until it stabilises in the working environment or as covered in the contract agreement.

➣ The PM will estimate the resource required for the warranty period and will retain only the resources essentially required during this period.

➣ All other resources are released as per project closure process of section 5.6 above.

➣ During the warranty period the effort spent in the project will be recorded in PCC or by using the time sheet.

➣ The issues addressed and releases made during the warranty period will be maintained using the configuration management processes by maintaining clear releases baselines.

➣ All resources will be released at the end of the warranty period is over and a formal project closure report will be sent to the customer by the PM.
