6.1 Purpose

The purpose of this Standard is to provide coding standard for C#.

6.2 Scope

This Standard is applicable for developing C# language based projects.

6.3 Naming Conventions

6.3.1 Capitalization Styles



Pascal Casing

The first letter of an identifier and the first letter of each subsequent concatenated word are capitalized. (eg: TestCounter)

Camel Casing

The first letter of an identifier is lowercase and the first letter of each subsequent concatenated word is capitalized. (eg: testCounter)

Uppercase

All letters in the identifier are capitalized only if it consists of an abbreviation

Which is one or two characters long (eg: PI) otherwise use Pascal casing instead.



6.3.2 Naming Guidelines



Use English for naming identifiers.

Do not use Hungarian notation. (In earlier days most of the programmers liked it - having the data type as a prefix for the variable name and using m_ as prefix for member variables. However, in .NET coding standards, this is not recommended. Usage of data type and m_ to represent member variables should not be used. All variables should use Camel Casing.)

Do use Camel Casing for member variables, local variables and parameters.

Do use Pascal Casing for property, method, event, class etc.

Do not use a prefix for identifiers( _ , m_ , s_ etc).

If you want to distinguish between local and member variables you should use “this”.

·        Do not contract words in identifiers, but do use well-known abbreviations.

Example:

Use GetWindow          /// Not GetWin

     Use UI                        /// Not UserInterface

·      Avoid using langauge specific terminology in names of identifiers.

Example:

Use void Write (double value); ///Not void Write (double doubleValue);

·        Name namespaces according to a well-defined pattern. The general rule is to use the company name fololwed by the technology name and optionally the feature and design.

CompanyName.TechnologyName.[.Feature][.Design]

·        Do not add suffix, prefix or underscore char to a class or struct name. Use a noun or noun phrase to name a class or struct.

·        Do prefix interface names with the letter I. Use a noun(IComponent),  a noun phrase(ICustomeAttributeProvider) or an adjective(IPersistable) to name an interface.

·        Use similar names when you define class/interface pair. For eg : a class implementing IComponent could be Component.

·        Do suffix attribute name with Attribute. /// For eg : ObsoleteAttribute.

·        Enumeration Names:

Do not add an Enum suffix to an enumeration type.

Use singular names for enumaration type.    /// For eg : Use Protocol instead of Protocols.Use

Use plural name for bit fields.                    /// For eg : SearchOptions

·        Name static fields with nouns, noun phrases or abbreviations for nouns.

·        Do use descriptive names, which should be enough to determine the variable meaning and it’s type.

·        Name methods with verbs or verb phrases.

·        Name properties using nouns or noun phrases. Consider naming a property with the same name as it’s type. 

·        Event Names:

1.       Use an EventHandler suffix on event handler names.

2.       Add Callback to delegates related to callback methods.

3.       Specify two parameters named sender and e.

4.       Name an event argument class with the EventArgs suffix.

5.       Consider naming events using a verb.

 

·        Suffix exception classes with Exception. For e.g. : ISomeException

·        Source File Name:

1.       Use Pascal casing for naming source files.

2.      Do not use the underscore character and do not use casing to differentiate names of files.

3.       Name the source file to the main class.

 

6.4 File Organization

·        Source files should contain only one public type, although multiple internal classes are allowed.

·        Directory names should follow the namespace for the class.

·        For e.g. : For MyProject.TestSuite.TestTier use MyProject/TestSuite/TestTier as the path, do not use the namespace name with dots.

·        Source files should be given the name of the public class in the file

·        Organize the material in each source file as follows:

1.       A file header

2.       using statements

3.       namespace statement, if appropriate

4.       A class, struct, or enum

 

 

6.5 Declarations

·       Declare and initialize variables close to where they are used.

·       If possible initialize variables at the point of declaration.

·       Declare each variable on a separate line.

·       Use const field to define constant values.

·        Use public static read-only fields for predefined object instances.

·       Avoid implementing destructor. If needed also use GC.SuperFinalize.

·       Do not access any reference in destructor since it is possible that references may be garbage collected and it may cause exception to be thrown.

 

 

6.6 Statements/Control Flow

·        Each line should contain only one statement.

·        All flow control primitives (if, else, for,while, do, switch) shall be followed by a block , even if it is empty.

·        All switch statements shall have a default label as a last case label.

·        Avoid multiple conditional return statements.

·        If, if-else, if else-if else Statements should look like this:

if (condition)

{

DoSomething();

...

} else if (condition)

{

DoSomethingOther();

...

} else

{

DoSomethingOtherAgain();

...

}

 

·        For / For each Statement should look like this:

for (initialization; condition; update)

{

- - - - - ;

}

 

·       While/do-while Statements should have the following form:

while (condition)

{

       ...

}

do

{

...

} while (condition);

 

 

·       Switch Statements should have the following form:

switch (condition)

{

case A:

...

break;

case B:

...

break;

default:

 ...

 break;

}

·        Try-catch Statements should have the following form:

try

{

...

} catch (Exception)

{

}

 

or

 

try

{

...

} catch (Exception e)

{

...

}

 

or

try

{

       ...

} catch (Exception e)

{

...

} finally

{

...

}

 

6.7 Indentation

The contents of all code blocks should be indented to improve readability. A single tab or four spaces are recommended as the standard indentation. Braces should be placed to show the level of indentation of the code block. Make it as readable as possible.

   int main()

   {

       doSomething()

   }

   struct myStruct

   {

        int x;

        int y;

   }

   if ( value == 0 )

   {

        doSomething();

   }

   else if ( value == 2 ) /* note position of cascaded if statement */

   {  

       doSomething2();

   }

   else

   {

       doSomething3();                          

   }

 

   while ( value < 300 )

   {

       doSomething();

   }

 

   do

   {

       doSomething();

   } while ( value < 300 )   /* ending brace and control on same line*/

 

   switch ( value )

   {

       case 1:

            doSomething();

              break;

        case 2:

        case 3:

           doSomething2();

           break;

       default:

           break;

   }

 

·       When an expression will not fit on a single line, break it up according to following principles:

1.      Break after a comma.

2.      Break after an operator.

3.       Align the new line with the beginning of the expression at the same level on the previous line.

 

 

Eg :

longMethodCall(expr1, expr2,
                     expr3, expr4, expr5);

 

var = a * b / (c - g + f) +         /// Avoid var = a * b / (c - g +
                    4 * z;                                                  f) + 4 * z;

 

·        Use SPACES for indentation. Do not use TABS. Tabs are interpreted differently in each software depending on the setting

·        Comments should be in the same level as the code.

·        Use one blank line to separate logical groups of code.

·        Braces ("{}") which enclose a block are to be placed in the same column, on separate lines directly before and after the block. The curly braces should be on a separate line and not in the same line as if, for etc.

 6.8  Comments and embedded documentation

·       All comments shall be written in US English.

·        Each file with must be documented with an introductory comment that provides information on the copy right information and description:

/******************************************************************

 

 Copyright information     : Imaje, Inc.

Brief description          : This file contains implementation of

                             ________.

******************************************************************/

·        When the code undergoes a modification, it should be preceded by documentation about the Revision Summary as shown below. The change code is repeated in the file where ever changes are done:

/*****************************************************************

                      Revision Summary

Date   -      change code   -      Description

*****************************************************************/

·        Add code comments in each file with XML tags, to that Code Comment Web Reports for each function/method of each file are created. The details of comments to be added are given below.

·        Every class should be documented with introductory comments that provide information about the class as shown:

/*******************************************************************

Class name           : TopicTemplate_i

Class type           : Concrete

Brief description    : Implementation class for the TopicTemplate IDL

Super classes        : TopicTemplateBOAImpl

Sub classes          : None

Classes used         : None

********************************************************************/

·        Every method should be documented with introductory comments that provide information about the method as shown:

/***********************************************************************

 

Name                 : createChampion

Synopsis             : gets the user details of a particular user.

Type                 : instance method

Input parameter      :      long topicId - description,

const char * championId - description,

environment IT_env - description

Output parameters    :  type param1 – description

Return Type          : Short - description

Global variables     :  var1, var2

Calling Syntax       : createChampion(topicId, championId);

*********************************************************************/

 

·       Use XML tags for documenting types and members.

·       Use #region to group non-public members.

 

6.9 Coding style

·        The public, protected, and private sections of a class or struct shall be declared in that order.

·        Write unary, increment, decrement, function call, subscript, and access operators together with their operands.

Eg :

      a  =  --b;   // not a = -- b;

·        Do not create source lines longer than 80 characters.

 

6.10 Miscellaneous

·        Declare all fields (data members) private.

·        Provide a default private constructor if there are only static methods and properties on a class.

·        All variants of an overloaded method shall be used for the same purpose and have similar behaviour.

·        Do not overload any ‘modifying’ operators on a class type.

In this context the ‘modifying’ operators are those that have a corresponding assignment operator, i.e. the nonunary versions of +, -, *, /, %, &, |, ^, << and >>.

Overloading operators on a struct type is good practice, since it is a value type.

·       If you implement one of operator==(), the Equals method or GetHashCode(), implement all three.

·       Also override this trio when you implement the IComparable interface.

·       Use a property rather than a method when the member is a logical data member.

·       Do not create a constructor that does not yield a fully initialized object.

·       Use explicit interface implementation only to prevent nameclashing or to support optional interfaces.

·       Exceptions :

1.      Do not throw exceptions from inside destructors.

2.      Use standard exceptions.

3.      Throw the most specific exception if possible.

·       Delegates :

1.      Do not make assumptions on the object’s state after raising an event.

2.      Raise events through protected virtual methods.

3.      Use the sender / arguments signature for event handlers.

·       Use an enum to strongly type parameters , properties and return types.

·       Do not use “magic numbers”.

·       Floating point values shall not be compared using either = = or != operators.

·       Do not cast types where a loss of precisions is possible.

·       Use the [Flags] attribute on an enum if a bitwise operation is to be performed on the numeric values.

·       When displaying error messages, in addition to telling what is wrong, the message should also tell what should the user do to solve the problem. Instead of message like "Failed to update database.", suggest what should the user do: "Failed to update database. Please make sure the login id and password are correct."

·       Show short and friendly message to the user. But log the actual error with all possible information. This will help a lot in diagnosing problems.
